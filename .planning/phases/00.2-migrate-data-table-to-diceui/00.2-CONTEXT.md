# Phase 0.2: Migrate Data Table to DiceUI - Context

**Gathered:** 2026-04-13
**Status:** Ready for planning

<domain>
## Phase Boundary

Replace the custom shadcn-based data-table infrastructure (13 components, 1 hook, 1 types file, 1 config file, 1 lib file) with @diceui/data-table components across all 9+ entity tables. Preserve existing server-side pagination, sorting, and filtering functionality. The deliverable is the same set of functional data tables powered by diceui instead of the custom component layer.

**What this phase is NOT:** Adding new table features, changing table behavior, adding new entity tables, or modifying the mock API layer. This is a like-for-like infrastructure swap.

</domain>

<decisions>
## Implementation Decisions

### Migration scope
- **D-01:** Component-by-component evaluation — for each of the 13 custom components in `components/data-table/`, check if @diceui/data-table provides a direct equivalent. If found, use the diceui component. If not found, keep the current component and wire it into diceui's composition.
- **D-02:** Accept diceui's default visual styling. Do not override diceui styles to match the old look. The only requirement is that diceui components inherit the current shadcn/radix-vega theme tokens (CSS custom properties) so the tables feel consistent with the rest of the app.

### Consumer table adaptation
- **D-03:** Keep a project-level wrapper component (updated `DataTableShell`) that wraps diceui's data-table with the standard toolbar, filter list, sort list, and empty state pattern. Consumer tables continue to pass columns + data + pageCount — minimal changes per consumer.
- **D-04:** Column definitions should remain in TanStack Table format (both the current implementation and diceui use `@tanstack/react-table`). Verify during research that diceui's column API is compatible. If diceui expects different `columnMeta` shape or additional config, update the column definitions across all consumers to match.

### URL state & hook strategy
- **D-05:** If @diceui/data-table provides its own hook or built-in URL state management (nuqs integration), adopt it and remove the custom `hooks/use-data-table.ts`. If diceui does not provide URL state management, keep the custom hook as fallback and wire it to diceui's rendering components.

### Cleanup
- **D-06:** Delete all replaced components, types, config, and lib files immediately after migration — no dead code. Components kept because diceui lacks equivalents remain in `components/data-table/`.
- **D-07:** Migrate table-by-table, one consumer at a time (e.g., vehicles first, then purchases, etc.). Each consumer migration is a separate atomic commit. This isolates issues and makes rollback granular.

### Claude's Discretion
- Migration order of the 12 consumer tables (pick the simplest table first as a proof-of-concept)
- How to structure the updated DataTableShell wrapper API
- Whether `config/data-table.ts` operator definitions need to change for diceui or can be kept as-is
- Whether `lib/data-table.ts` utilities (column pinning, filter helpers) are still needed or replaced by diceui equivalents
- Whether `types/data-table.ts` module augmentation needs updating for diceui's expected ColumnMeta shape
- Handling of the `lib/parsers.ts` sorting state parser — keep if diceui doesn't provide equivalent

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- **DataTableShell pattern:** The wrapper pattern (columns + data + pageCount → full table with toolbar) is well-established across 12 consumers. The updated wrapper should preserve this API shape to minimize consumer changes.
- **Column definition files:** Each entity has a `*-columns.tsx` file with TanStack Table column defs including `columnMeta` for filter variants and options. These are the primary files that need verification against diceui's expected format.
- **EmptyState component:** `components/shared/empty-state.tsx` is used by the shell for no-results display. This is independent of the data-table infrastructure and should be kept.

### Established Patterns
- **nuqs URL state sync:** All table state (page, perPage, sort, filters, joinOperator) synced to URL params via nuqs. diceui also uses nuqs, so URL param conventions should align.
- **Manual server-side operations:** All tables use `manualPagination: true`, `manualSorting: true`, `manualFiltering: true` — data comes from mock Route Handlers. diceui must support this pattern.
- **Column pinning:** `lib/data-table.ts` provides `getColumnPinningStyle()` for sticky columns. Check if diceui handles this natively.

### Integration Points
- **Mock Route Handlers:** `/api/*` handlers return `{ data, total, page, pageSize }` shape. Tables consume this via TanStack Query. The API layer is NOT changing — only the table rendering layer.
- **TanStack Query:** Consumer tables fetch data via `useQuery`/`useInfiniteQuery` and pass results to the table. This integration stays the same.
- **Sheet dialogs:** Several pages use Sheet-based forms triggered from table row actions. The row action pattern (via `DataTableRowAction` type) needs to work with diceui's row rendering.

</code_context>
