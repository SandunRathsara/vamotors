# Phase 00.2: Code Review Report

**Reviewed:** 2026-04-13
**Depth:** standard
**Files Reviewed:** 21
**Status:** issues_found

## Summary

These files form the complete data table layer for VA Motors — a set of column definitions and table shell wrappers that feed `DataTableShell` (backed by TanStack Table v8). The architecture is consistent and well-structured overall: column factories, query-driven server pagination, and nuqs-based URL state are all applied uniformly across domains.

One critical issue was found: a hardcoded customer ID in the lease deal creation payload that will silently send wrong data to the API. Four warnings cover a stale-closure suppression, an un-memoized column factory causing unnecessary re-renders, a REST endpoint mismatch for user deletion, and module-scope fixture maps that will break silently when real API data replaces mock data. Four info items cover dead code and cross-cutting duplication.

## Critical Issues

### CR-01: Hardcoded `customerId` in Lease Deal Creation

**File:** `components/lease/lease-deals-table.tsx:77`
**Issue:** The `handleCreateDeal` submission payload always sends `customerId: "c-001"` regardless of which user is logged in or which customer the deal should belong to. There is no customer selection field in the form. When this component is wired to a real API, every lease deal created from the UI will be attributed to the same hardcoded fixture customer.
**Fix:** Either add a customer selection field to the form and use `fd.get("customerId")`, or derive the customer ID from the authenticated session:

```tsx
// Option A: add a customer field to the form
const payload = {
  customerId: fd.get("customerId") as string,
  // ...
}

// Option B: derive from auth session (if deals always belong to the acting customer)
const { data: session } = useSession()
const payload = {
  customerId: session?.user?.id ?? "",
  // ...
}
```

## Warnings

### WR-01: Un-memoized Column Factory Recreated Every Render

**File:** `components/lease/reconciliation-table.tsx:196-199`
**Issue:** `buildColumns` is called directly inside the render body without `useMemo`. TanStack Table uses referential equality to detect column changes — when the columns array is a new object reference on every render, the table re-initializes its internal state on every render cycle (including after the `receivedAmount` input state changes). This can cause subtle flicker and lost scroll/selection state.

```tsx
// Current — recreated on every render
const columns = buildColumns((item) => {
  setSelectedItem(item)
  setReceivedAmount(String(item.expectedCommission))
})
```

**Fix:** Wrap in `useMemo`:

```tsx
const handleReconcile = React.useCallback((item: LeaseReconciliation) => {
  setSelectedItem(item)
  setReceivedAmount(String(item.expectedCommission))
}, [])

const columns = React.useMemo(
  () => buildColumns(handleReconcile),
  [handleReconcile],
)
```

### WR-02: Stale Closure Suppressed with eslint-disable in ThirdPartiesTable

**File:** `components/third-parties/third-parties-table.tsx:81-85`
**Issue:** `columns` is memoized with an empty dependency array `[]`, but `handleArchive` (used inside `createThirdPartiesColumns`) is not listed as a dependency. The `react-hooks/exhaustive-deps` rule is explicitly suppressed to hide this. If `handleArchive` ever captures variables from the outer closure that change (e.g., derived from props), the action button will silently call the stale version. The suppression comment also prevents ESLint from catching future regressions.

```tsx
const columns = React.useMemo(
  () => createThirdPartiesColumns(handleArchive),
  // eslint-disable-next-line react-hooks/exhaustive-deps
  [],
)
```

**Fix:** Stabilize `handleArchive` with `useCallback` and add it as a dependency:

```tsx
const handleArchive = React.useCallback((tp: ThirdParty) => {
  setArchiveTarget(tp)
}, [])

const columns = React.useMemo(
  () => createThirdPartiesColumns(handleArchive),
  [handleArchive],
)
```

### WR-03: User Deletion Targets Collection Endpoint Instead of Resource Endpoint

**File:** `components/users/users-columns.tsx:48-53`
**Issue:** `DeleteUserCell` calls `useEntityMutation` with `endpoint: "/api/users"` and `method: "PATCH"`, then sends `{ id: user.id, status: "Inactive" }`. This hits the collection endpoint rather than `/api/users/:id`. REST convention requires mutations on a specific resource to target that resource's URL. When the real API is wired, the PATCH to `/api/users` will either fail or affect the wrong record depending on server-side routing.

**Fix:**

```tsx
const { mutate, isPending } = useEntityMutation({
  entityKey: "users",
  endpoint: `/api/users/${user.id}`,
  method: "PATCH",
  successMessage: "User deleted.",
})

function handleDelete() {
  mutate(
    { status: "Inactive" } as Parameters<typeof mutate>[0],
    { onSuccess: () => setOpen(false) },
  )
}
```

### WR-04: Module-Scope Fixture Lookup Maps Will Silently Return Wrong Data After API Migration

**File:** `components/customers/customers-columns.tsx:19-30`, `components/repairs/repairs-columns.tsx:22-23`, `components/sales/sales-columns.tsx:24-25`
**Issue:** Three column files build lookup Maps at module scope from imported fixture arrays (`salesFixtures`, `vehicleFixtures`, `thirdPartiesFixtures`, `customerFixtures`). These maps are computed once when the module is first imported and never updated. When real API data replaces fixtures, these columns will silently show stale counts, names, and dates sourced from fixture data rather than live data — and there will be no runtime error to alert the developer.

This is an acceptable tradeoff during the mock-data phase but represents a migration hazard. The pattern needs to be replaced before connecting to real API endpoints.

**Fix when real API is wired:** Remove module-scope maps. Instead, either (a) denormalize the join server-side and return it in the API response, or (b) fetch related entity data via TanStack Query inside the column cell and use `queryClient.getQueryData` to read from cache:

```tsx
// Preferred: denormalize on server
// API returns: { ...repair, vehicleName: "Toyota Axio 2020", vendorName: "ABC Motors" }

// Column cell becomes:
cell: ({ row }) => <span>{row.original.vehicleName}</span>
```

## Info

### IN-01: Dead Prop — `searchPlaceholder` Declared But Never Used

**File:** `components/shared/data-table-shell.tsx:31`
**Issue:** `searchPlaceholder?: string` is declared in `DataTableShellProps` and documented in a JSDoc comment, but the implementation never reads or forwards this value. No consumer passes it either. It is a dead interface member that adds noise to the type contract.
**Fix:** Remove the prop declaration from the interface, or implement it by passing it to a search `Input` inside the toolbar if that is the intended design.

### IN-02: `STATUS_OPTIONS` Exported But Apparently Unused

**File:** `components/lease/lease-deals-table.tsx:199-200`
**Issue:** `STATUS_OPTIONS` is declared at module scope and re-exported at the bottom of the file, but it is not used anywhere in the component itself (the form uses a hardcoded `"Initiated"` default and does not render a status selector). If no other file imports it, this is dead code.
**Fix:** Remove `STATUS_OPTIONS` if it is not consumed elsewhere, or use it in the form to replace the hardcoded `status: "Initiated"` value.

### IN-03: `formatDate` Helper Duplicated Across Five Files

**Files:** `components/approvals/approvals-columns.tsx:14-21`, `components/customers/customers-columns.tsx:32-39`, `components/repairs/repairs-columns.tsx:25-32`, `components/lease/reconciliation-table.tsx:35-42`, `components/third-parties/third-parties-columns.tsx:38-45`
**Issue:** An identical `formatDate(dateStr?: string): string` function using `date-fns/format` with a `dd MMM yyyy` pattern and a try/catch fallback is copy-pasted into five separate files. Any change to the date format or error handling must be replicated in all five places.
**Fix:** Extract to a shared utility, e.g. `lib/format.ts`:

```ts
import { format } from "date-fns"

export function formatDate(dateStr?: string, pattern = "dd MMM yyyy"): string {
  if (!dateStr) return "—"
  try {
    return format(new Date(dateStr), pattern)
  } catch {
    return dateStr
  }
}
```

### IN-04: Reconcile Sheet Pre-fills Received Amount with Expected Amount

**File:** `components/lease/reconciliation-table.tsx:198`
**Issue:** When the reconcile action is triggered, `receivedAmount` is pre-populated with `String(item.expectedCommission)`. In a commission reconciliation workflow the received amount is almost certainly different from the expected amount (otherwise there would be no discrepancy to reconcile). The pre-fill creates false confidence and requires the user to always clear the field before entering the real value.
**Fix:** Pre-fill with an empty string, or if pre-filling is desired for convenience, make the field clearly labelled as a suggestion:

```tsx
setReceivedAmount("") // user enters actual received amount
```
