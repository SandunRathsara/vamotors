## Summary

**The critical discovery of this research: the project already has the diceui data-table infrastructure installed.** The existing `components/data-table/` directory (13 components), `hooks/use-data-table.ts`, `config/data-table.ts`, `types/data-table.ts`, `lib/data-table.ts`, and `lib/parsers.ts` are all identical (or substantively equivalent) to what `pnpm dlx shadcn@latest add @diceui/data-table` would install. This is not a coincidence — the project appears to have been originally populated from the same sadmann7/diceui source.

The real migration work is therefore in the **consumer layer**. All 12 consumer tables (`vehicles-table.tsx`, `purchases-table.tsx`, etc.) currently bypass the diceui `useDataTable` hook entirely. Each consumer implements its own ad-hoc URL state using raw `useQueryStates` with manually named params (`page`, `pageSize`, `q`, `status`, etc.) and passes data directly to `DataTableShell`. The `DataTableShell` then calls `useDataTable` but with no column `enableColumnFilter` or `meta.variant` set, so the filter/sort UX (the `DataTableFilterList` and `DataTableSortList` components that are already installed) is completely idle — the toolbar renders but filter/sort buttons are non-functional.

**The migration work is:** (1) verify the installed components match the current diceui spec, (2) wire each consumer to actually use `useDataTable` properly by adding `enableColumnFilter` and `meta` to column definitions, (3) remove the consumer-level `useQueryStates` duplication, (4) update `DataTableShell` to pass query keys through, and (5) delete now-unused code. No new npm packages are needed. NuqsAdapter is already configured.

**Primary recommendation:** Treat this as wiring, not replacement. The hard work (component implementation) is done. Each consumer migration is: add `meta` to columns → remove manual `useQueryStates` → let `useDataTable` own URL state → verify filter/sort UI works → commit.

### What the diceui `useDataTable` hook provides (that consumers currently ignore)

[VERIFIED: `hooks/use-data-table.ts` — read directly in this session]

The hook manages ALL URL state for the table:
- `page` / `perPage` — pagination via `parseAsInteger`
- `sort` — sorting state via `getSortingStateParser` (JSON array in URL)
- `filters` — advanced filter list via `getFiltersStateParser` (JSON array in URL)
- `joinOperator` — AND/OR for filter combination

When `enableAdvancedFilter: true`, the hook disables the simple `filterValues` path and defers to `DataTableFilterList` which manages its own `filters` + `joinOperator` query state directly.

Consumer tables currently define their own URL params (`page`, `pageSize`, `q`, `status`, `sortBy`, `sortDir`) then pass raw data to `DataTableShell`. The shell calls `useDataTable` but with no column filter config, so the filter UI buttons render as no-ops.

### How diceui ColumnMeta drives the filter UI

[VERIFIED: `types/data-table.ts` + `components/data-table/data-table-filter-list.tsx` — read directly]

The `DataTableFilterList` reads `column.columnDef.meta?.variant` to determine which filter input to render. The full `ColumnMeta` shape already declared in `types/data-table.ts`:

```typescript
interface ColumnMeta<TData, TValue> {
  label?: string;         // Display name shown in filter field selector
  placeholder?: string;   // Input hint text
  variant?: FilterVariant; // "text" | "number" | "range" | "date" | "dateRange" | "boolean" | "select" | "multiSelect"
  options?: Option[];     // Required when variant is "select" or "multiSelect"
  range?: [number, number]; // Required when variant is "range"
  unit?: string;          // Suffix for numeric display
  icon?: React.FC<...>;   // Column icon
}
```

**No ColumnMeta changes needed** — the shape in the project matches diceui exactly.

Columns that should be filterable need: `enableColumnFilter: true` plus a `meta.variant` and `meta.label`. Columns currently have `enableSorting: true/false` set correctly but are missing `enableColumnFilter` and `meta`. This is the primary column definition update needed per consumer.

### Consumer URL state alignment

**Problem:** After migration, consumers need to read URL params to drive the API call. `useDataTable` owns params named `page`, `perPage`, `sort`, `filters`, `joinOperator`. Current consumers use `page`, `pageSize`, `sortBy`, `sortDir`.

**Solution:** Each consumer reads the same nuqs keys that `useDataTable` writes. The key names are configurable via `queryKeys` prop. Default key names from the hook:
- `page` → OK, same as current
- `perPage` → consumers currently use `pageSize` — needs to change to `perPage`
- `sort` → JSON array `[{id:"makeModel",desc:false}]` — replaces separate `sortBy` + `sortDir`
- `filters` → JSON array of filter objects — replaces separate `q`, `status` etc.

Each consumer's API call handler must switch from reading `sortBy`/`sortDir` to parsing the `sort` URL param. The mock Route Handlers also need to handle this updated shape — but per CONTEXT.md the API layer is NOT changing. **Resolution:** Consumer tables can parse the `sort` JSON param client-side and map it back to `sortBy`/`sortDir` for the API call, or the `DataTableShell` can expose parsed sort state via a callback or `queryKeys` override.

**Simpler resolution:** Override `queryKeys` in `DataTableShell` to use the existing key names where possible, or accept a `queryKey` prefix to namespace tables on pages with multiple tables.

## Common Pitfalls

### Pitfall 1: Duplicate URL state — consumer + shell both owning the same params

**What goes wrong:** Consumer defines `useQueryStates({ page, pageSize })` and `DataTableShell` also calls `useDataTable` which writes `page`, `perPage`. Two writers for the same URL param cause race conditions and stale reads.

**Why it happens:** Current consumers write URL state themselves before passing data to the shell. After migration, the shell owns URL state. If consumer code is not cleaned up, both run simultaneously.

**How to avoid:** In each consumer migration, remove ALL `useQueryStates` / `useQueryState` calls that duplicate keys managed by `useDataTable`. The consumer should read from the table instance (returned by the hook) or from the same nuqs keys the hook writes.

**Warning signs:** URL params flicker on first render, or pagination/sort resets unexpectedly.

### Pitfall 2: `pageSize` vs `perPage` URL key mismatch

**What goes wrong:** Mock Route Handlers are called with `pageSize` but `useDataTable` writes `perPage`. API returns wrong page count, pagination breaks.

**Why it happens:** `useDataTable` defaults to `perPage` as the URL key. Current consumers use `pageSize`. The mock API is wired to `pageSize`.

**How to avoid:** Either (a) pass `queryKeys: { perPage: "pageSize" }` to `useDataTable` via `DataTableShell` to override the URL key name, or (b) update Route Handlers to accept `perPage` — but CONTEXT.md says the API layer is NOT changing. Use option (a).

**Warning signs:** Page count always shows 1 page, or `perPage` in URL but API is called with `perPage=undefined`.

### Pitfall 3: Sort state format mismatch — JSON array vs `sortBy`/`sortDir` string params

**What goes wrong:** `useDataTable` writes `sort=[{"id":"makeModel","desc":false}]` to the URL. Mock Route Handlers read `sortBy` and `sortDir` as separate string params. Sort stops working.

**Why it happens:** The diceui hook uses a structured sort format to support multi-column sorting. Current mock APIs use the simpler two-param approach.

**How to avoid:** In the consumer's `useEntityQuery` call, parse the `sort` URL param and map it to `sortBy`/`sortDir`. Use `useQueryState("sort", getSortingStateParser())` in the consumer to read the typed sort array and extract the first sort field:

```typescript
const [sorting] = useQueryState("sort", getSortingStateParser().withDefault([]))
const sortBy = sorting[0]?.id ?? ""
const sortDir = sorting[0]?.desc ? "desc" : "asc"
```

**Warning signs:** Table shows sorted UI state but API returns unsorted data.

### Pitfall 4: Columns without `meta.label` show undefined in FilterList

**What goes wrong:** `DataTableFilterList` displays `undefined` instead of a column name in the filter field selector.

**Why it happens:** `DataTableSortList` and `DataTableFilterList` read `column.columnDef.meta?.label` for display. Current column definitions don't set `meta.label`.

**How to avoid:** Add `meta: { label: "Make / Model" }` (or similar) to every column that has `enableSorting: true` or `enableColumnFilter: true`. Even if the filter UI isn't used yet, label is needed for sort list display.

**Warning signs:** Sort list button shows items with blank labels.

### Pitfall 5: `DataTableShell` calling `useDataTable` with `pageSize: 20` hardcoded overrides URL state

**What goes wrong:** The shell passes `initialState: { pagination: { pageIndex: 0, pageSize: 20 } }`. After migration, consumers might set a different page size via URL. The `pageSize: 20` initial state overrides the URL's `perPage` param on first render.

**Why it happens:** `initialState` in `useReactTable` only applies if there's no existing URL state. But the hook uses `withDefault(initialState?.pagination?.pageSize ?? 10)` for `perPage` — so the default IS the initial page size, and URL overrides it correctly.

**Actual risk:** Low. nuqs `withDefault` properly defers to URL when present. But verify this during the vehicles proof-of-concept.

### Pitfall 6: Tables on pages with multiple tables share the same URL params

**What goes wrong:** Two tables on the same page (e.g., reconciliation page with multiple lease-type tables) both use `page=1&perPage=20` — the same URL params — causing them to interfere.

**Why it happens:** `useDataTable` uses global URL param names by default.

**How to avoid:** Pass `queryKeys` prefix to `DataTableShell` for pages that need namespacing. Example: `queryKeys: { page: "leaseDealsPage", perPage: "leaseDealsPerPage" }`. The lease/reconciliation tables may need this.

**Warning signs:** Paginating one table on the page changes the page of the other table.

### DataTableShell consuming diceui hook properly

```typescript
// Source: hooks/use-data-table.ts + components/data-table/data-table.tsx
export function DataTableShell<TData>({
  columns,
  data,
  pageCount,
  emptyState,
  toolbarChildren,
}: DataTableShellProps<TData>) {
  const { table } = useDataTable({
    data,
    columns,
    pageCount,
    initialState: {
      pagination: { pageIndex: 0, pageSize: 20 },
    },
    enableAdvancedFilter: false,  // basic mode: filter list + sort list available
  })

  const isEmpty = data.length === 0

  return (
    <DataTable table={table}>
      <DataTableAdvancedToolbar table={table}>
        {toolbarChildren}
        <DataTableFilterList table={table} shallow={false} />
        <DataTableSortList table={table} />
      </DataTableAdvancedToolbar>
      {isEmpty && (
        <div className="min-h-[300px] flex items-center justify-center">
          <EmptyState ... />
        </div>
      )}
    </DataTable>
  )
}
```

### Consumer reading URL state for API call after migration

```typescript
// Source: lib/parsers.ts getSortingStateParser, hooks/use-data-table.ts key names
export function VehiclesTable() {
  // Read the same URL keys that useDataTable writes
  const [page] = useQueryState("page", parseAsInteger.withDefault(1))
  const [perPage] = useQueryState("perPage", parseAsInteger.withDefault(20))
  const [sorting] = useQueryState("sort", getSortingStateParser<Vehicle>().withDefault([]))

  const sortBy = sorting[0]?.id ?? ""
  const sortDir = sorting[0]?.desc ? "desc" : "asc"

  const { data } = useEntityQuery<PaginatedResponse<Vehicle>>(
    "vehicles",
    "/api/vehicles",
    { page, pageSize: perPage, sortBy: sortBy || undefined, sortDir },
  )

  return (
    <DataTableShell
      columns={vehicleColumns}  // meta.label set, enableColumnFilter where needed
      data={data?.data ?? []}
      pageCount={Math.max(1, Math.ceil((data?.total ?? 0) / perPage))}
    />
  )
}
```
