# Phase 0.3: Testing Strategy - Research

**Researched:** 2026-04-15
**Domain:** Three-tier test harness (Vitest + Playwright) + `just` task runner for Next.js 16 / React 19 monolith
**Confidence:** HIGH

## Summary

Phase 0.3 builds the test platform, not the tests. It installs three tiers — Vitest+jsdom+RTL for unit/component, `playwright.request` for API, Playwright browser context for E2E — plus a `Justfile` that wires pnpm scripts into a consistent developer surface. One example test ships per tier (plus a fourth component example per D-20) to prove each wire-up against the existing Phase 0.1 mock shell; the component example targets a DiceUI column definition from Phase 0.2.

All four base libraries have first-class, current support for the exact stack in play (Next.js 16.1.7, React 19.2.5, pnpm 10, TypeScript 5.9, Tailwind v4.2): Vitest 4.1.4 with `@vitejs/plugin-react` 6.0.1 handles JSX/TSX through SWC-free Vite; `@testing-library/react` 16.3.2 is the React-19-compatible line; Playwright 1.59.1's `webServer` + `projects` pattern cleanly produces two projects (API + E2E) sharing one `next build && next start` and one `baseURL`; `just` 1.49 is already installed on the dev machine and its attribute grammar covers recipe grouping, platform guards, and default-recipe-as-help cleanly.

**Primary recommendation:** Pin exact versions (matching the existing `package.json` convention from Phase 0.1 D-02), install `@vitejs/plugin-react` (do NOT rely on Next.js SWC — Vitest runs Vite, not Next), use Playwright's dependency-based project model (`setup` project produces `.auth/user.json`, `e2e` and `api` projects consume it via `storageState`), and keep the dev-only reset endpoint implemented today against `lib/mock-store.ts` with a Prisma-swap seam for Phase 1+.

## User Constraints (from CONTEXT.md)

### Locked Decisions

**Test tiers and boundaries**
- **D-01:** Three tiers: (a) unit/component via **vitest + jsdom + @testing-library/react**, (b) API via **@playwright/test** using `playwright.request`, (c) E2E via **@playwright/test** browser context.
- **D-02:** Route Handlers are tested in both tiers with a split by concern: pure logic (Zod parsing, business rules, calculations) is extracted into `lib/` and unit-tested; the Route Handler file itself is exercised only through the API tier over real HTTP.
- **D-03:** Component tier covers logic-heavy components only — hooks, reducers, form validation, data-table column defs, utility components. shadcn primitive wrappers and purely presentational components are NOT unit-tested; they're covered transitively by E2E.
- **D-04:** Out of scope for this phase: perf/load testing, accessibility automation (axe-core/a11y), visual regression (screenshot diffing), cross-browser matrix — Chromium only for E2E.

**Just recipe surface**
- **D-05:** `just test` runs all three tiers in sequence (unit → api → e2e).
- **D-06:** `just test <feature>` uses **path-prefix filtering** — runs any test file whose path contains the argument, across all tiers. No tag decorators.
- **D-07:** `just check` is the fast pre-commit recipe: typecheck (`tsc --noEmit`), lint (`eslint`), format verify (`prettier --check`), and unit tier only (vitest). Playwright tiers excluded to keep `just check` fast.
- **D-08:** `just ci` is the single recipe CI invokes: `just check` + api tier + e2e tier + `pnpm build`.
- **D-09:** `just test-e2e-ui` opens Playwright UI mode for interactive E2E debugging.
- **D-10:** No `just test-watch` and no `just test-file` recipes — user explicitly did not request them.
- **D-11:** `just coverage` generates v8 coverage reports for the unit tier. No CI-enforced thresholds.

**E2E and API infrastructure**
- **D-12:** E2E runs against a **production build** — Playwright `webServer` config does `next build && next start`. Cold start ~30s acceptable. API tier reuses the same webServer.
- **D-13:** API tier uses `playwright.request` against the running server — no direct handler imports at this tier.
- **D-14:** Dev-only reset endpoint `POST /api/__test__/reset` guarded by NODE_ENV != production. Tests call in `beforeEach`. Rehydrates the in-memory mock-data store in Phase 0.3; Prisma truncation in later phases. Endpoint contract stays stable.
- **D-15:** Playwright `storageState` per role. Phase 0.3 scaffolds `globalSetup` + `.auth/` directory and `.gitignore`s it. Actual login wired in Phase 1.

**File layout**
- **D-16:** Unit + component tests colocated as `*.test.ts(x)` next to source.
- **D-17:** E2E in `tests/e2e/`, API in `tests/api/`.

**Coverage**
- **D-18:** v8 coverage, no thresholds. `just coverage` produces HTML + text reports.

**Example tests to ship**
- **D-19:** Unit example — integer-cents money/currency helper (`lib/money.ts` or equivalent).
- **D-20:** Component example — one Phase 0.2 DiceUI data-table column definition test (sort/filter logic).
- **D-21:** API example — `playwright.request` hits one existing Phase 0.1 mock Route Handler.
- **D-22:** E2E example — one golden-path navigation flow against mock data.

### Pattern 1: Vitest config for Next.js 16 + React 19

**What:** A single `vitest.config.ts` at the repo root that uses `@vitejs/plugin-react`, jsdom environment, colocated `*.test.ts(x)` glob, path alias matching Next.js's `tsconfig.json` `paths`, and a setup file that loads `@testing-library/jest-dom/vitest`.

**When to use:** Every unit and component test. Vitest is invoked via `pnpm exec vitest run` (no watch for CI/just check) and `pnpm exec vitest` for watch (manual, per D-10).

**Example:**
```ts
// Source: /vitest-dev/vitest docs (ctx7, 2026-04-15)
// vitest.config.ts
import { defineConfig } from 'vitest/config';
import react from '@vitejs/plugin-react';
import { fileURLToPath } from 'node:url';

export default defineConfig({
  plugins: [react()],
  resolve: {
    alias: {
      '@': fileURLToPath(new URL('./', import.meta.url)),
    },
  },
  test: {
    environment: 'jsdom',
    globals: false,                         // prefer explicit `import { expect } from 'vitest'`
    include: ['**/*.test.{ts,tsx}'],        // colocated per D-16
    exclude: ['**/node_modules/**', 'tests/e2e/**', 'tests/api/**', '.next/**'],
    setupFiles: ['./vitest.setup.ts'],
    clearMocks: true,
    restoreMocks: true,
    coverage: {
      provider: 'v8',                       // D-18
      reporter: ['text', 'html'],
      include: ['lib/**', 'components/**', 'hooks/**'],
      exclude: ['**/*.test.{ts,tsx}', '**/*.d.ts', 'lib/mock-data/**'],
    },
  },
});
```

```ts
// vitest.setup.ts
import '@testing-library/jest-dom/vitest';
import { cleanup } from '@testing-library/react';
import { afterEach } from 'vitest';

afterEach(() => {
  cleanup();
});
```

**Notes verified against ctx7 docs:**
- `environment: 'jsdom'` is the correct key (ctx7 Vitest docs).
- `globals: false` is fine — Vitest's matchers and helpers are importable from `'vitest'`.
- `@testing-library/jest-dom/vitest` is the official subpath that registers matchers with Vitest's `expect` — do not use the bare `@testing-library/jest-dom` import (that targets Jest).
- Vitest excludes must include `tests/e2e/**` and `tests/api/**` so Playwright specs don't accidentally get collected by the unit runner.

### Pattern 2: Playwright config with webServer + three projects

**What:** A single `playwright.config.ts` with one `webServer` entry running `next build && next start` and three projects: `setup` (scaffold for `.auth/`), `api` (no browser chrome — uses `request` fixture), and `e2e` (Chromium). `api` and `e2e` both `dependencies: ['setup']`.

**When to use:** API and E2E tiers, invoked as `pnpm exec playwright test --project=api`, `--project=e2e`, or no filter for both.

**Example:**
```ts
// Source: /microsoft/playwright docs (ctx7, 2026-04-15) — webServer + projects + storageState + request
// playwright.config.ts
import { defineConfig, devices } from '@playwright/test';

const PORT = Number(process.env.PORT ?? 3000);
const baseURL = `http://localhost:${PORT}`;

export default defineConfig({
  testDir: 'tests',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 1 : 0,
  workers: process.env.CI ? 1 : undefined,
  reporter: [['list'], ['html', { open: 'never' }]],

  // D-12: production build, shared by api + e2e projects.
  webServer: {
    command: `pnpm build && pnpm start -p ${PORT}`,
    url: baseURL,
    reuseExistingServer: !process.env.CI,
    stdout: 'pipe',
    stderr: 'pipe',
    timeout: 120_000,  // ~30s cold start budget + headroom; D-12 accepts ~30s
  },

  use: {
    baseURL,
    trace: 'on-first-retry',
    extraHTTPHeaders: {
      Accept: 'application/json',
    },
  },

  projects: [
    // D-15: scaffold-only in Phase 0.3. Phase 1 fills in real login steps.
    { name: 'setup', testMatch: /.*\.setup\.ts$/ },

    // D-13: API tier via playwright.request, no browser context needed.
    {
      name: 'api',
      testDir: 'tests/api',
      dependencies: ['setup'],
      use: {
        // Once Phase 1 lands, swap in role-specific auth state:
        // storageState: 'playwright/.auth/user.json',
      },
    },

    // E2E tier — Chromium only (D-04).
    {
      name: 'e2e',
      testDir: 'tests/e2e',
      dependencies: ['setup'],
      use: {
        ...devices['Desktop Chrome'],
        // storageState: 'playwright/.auth/user.json', // enable in Phase 1
      },
    },
  ],
});
```

**Notes verified against ctx7 docs:**
- `webServer` accepts `command`, `url`, `reuseExistingServer`, and `timeout` exactly as above (Playwright docs).
- The `projects[].dependencies` mechanism is the canonical replacement for the older single-`globalSetup` pattern when you want multiple projects sharing prepared state.
- `storageState` at `use.storageState` reads the file produced by the `setup` project.
- `testDir` per-project is the cleanest way to enforce D-17's `tests/e2e/` vs `tests/api/` split without naming collisions.
- A single `playwright.config.ts` (not two separate configs) is idiomatic; the `api` vs `e2e` distinction is a project, not a second config.

### Pattern 3: Playwright `request` fixture for the API tier

**What:** Use the per-test `request` fixture (no browser) to hit the running webServer over real HTTP.

**Example:**
```ts
// Source: /microsoft/playwright docs (ctx7, 2026-04-15) — api-testing-js
// tests/api/vehicles.api.spec.ts
import { test, expect } from '@playwright/test';

test.beforeEach(async ({ request }) => {
  // D-14: reset the in-memory mock store before every test
  const res = await request.post('/api/__test__/reset');
  expect(res.ok()).toBe(true);
});

test('GET /api/vehicles returns paginated shape', async ({ request }) => {
  const res = await request.get('/api/vehicles?page=1&pageSize=10');
  expect(res.status()).toBe(200);

  const body = await res.json();
  expect(body).toMatchObject({
    data: expect.any(Array),
    total: expect.any(Number),
    page: 1,
    pageSize: 10,
  });
  expect(body.data.length).toBeLessThanOrEqual(10);
});
```

### Pattern 4: Dev-only reset endpoint (D-14)

**What:** A Route Handler at `app/api/__test__/reset/route.ts` that returns 404 in production and rehydrates the in-memory store in dev/test.

**Why this shape:** Next.js 16 App Router does not support build-time exclusion of a route by environment — the handler file is compiled either way. The idiomatic gate is a runtime check that returns `notFound()` or a 404 Response when `NODE_ENV === 'production'`. This keeps the contract stable for Phase 1+ (swap the body to Prisma truncation; Route Handler wrapper stays the same).

**Example:**
```ts
// Source: Next.js App Router docs (training-data, cross-checked against app/api/** structure) [CITED: nextjs.org/docs/app/api-reference/file-conventions/route]
// app/api/__test__/reset/route.ts
import { NextResponse } from 'next/server';
import { resetAll } from '@/lib/mock-store'; // to be added: exports a function that rehydrates every Map

export const dynamic = 'force-dynamic';

export async function POST() {
  if (process.env.NODE_ENV === 'production') {
    return new NextResponse('Not Found', { status: 404 });
  }
  resetAll();
  return NextResponse.json({ ok: true });
}
```

**Mock-store seam:** `lib/mock-store.ts` currently exposes per-entity Maps. The planner should add a `resetAll()` function that re-reads the Phase 0.1 fixtures into those Maps. Phase 1's Prisma version replaces the function body with `await prisma.$transaction([prisma.<table>.deleteMany(), ...])`; the Route Handler wrapper does not change.

**Security:** Return `Response` for a hard 404, not `notFound()` (the latter renders a UI page for GET but gives inconsistent behavior for POST). The NODE_ENV gate is sufficient for the mock phase; when Prisma lands, add a second gate on a feature-flag env var (e.g., `TEST_RESET_ENABLED=1`) so even preview deployments can't accidentally truncate.

### Pattern 6: Justfile — full recipe surface for D-05..D-11

```just
# Justfile
# Source: /casey/just docs (ctx7, 2026-04-15) — attributes, grouping, default recipe as help
set shell := ["bash", "-cu"]
set dotenv-load := true
export FORCE_COLOR := "1"

# Default recipe: show the list.
default:
    @just --list

# ---- D-07 fast check ----
[group('check')]
check: typecheck lint format-check test-unit

[group('check')]
typecheck:
    pnpm typecheck

[group('check')]
lint:
    pnpm lint

[group('check')]
format-check:
    pnpm exec prettier --check "**/*.{ts,tsx,md,json}"

# ---- D-05 / D-19..D-22 test tiers ----
# D-05: `just test` runs all three tiers sequentially.
# D-06: `just test <feature>` filters by path substring across all tiers.
[group('test')]
test feature="":
    just test-unit {{feature}}
    just test-api {{feature}}
    just test-e2e {{feature}}

[group('test')]
test-unit feature="":
    pnpm exec vitest run {{feature}}

[group('test')]
test-api feature="":
    pnpm exec playwright test --project=api {{feature}}

[group('test')]
test-e2e feature="":
    pnpm exec playwright test --project=e2e {{feature}}

# D-09: interactive debugger for E2E
[group('test')]
test-e2e-ui:
    pnpm exec playwright test --project=e2e --ui

# D-11: coverage for unit tier, no thresholds
[group('test')]
coverage:
    pnpm exec vitest run --coverage

# ---- D-08 CI gate ----
[group('ci')]
ci: check test-api test-e2e
    pnpm build
```

**Verified facts against ctx7 `just` docs:**
- `default: @just --list` is the idiomatic help-as-default pattern.
- `[group('name')]` attribute groups recipes in `--list` output.
- Recipe dependencies (`check: typecheck lint …`) execute in order, stopping on first failure — perfect for D-07.
- Parameters with defaults (`feature=""`) are the mechanism for D-06's optional path filter; `{{feature}}` expands empty-safely. Vitest and Playwright both accept a trailing positional path/substring argument and ignore empty strings.
- `[confirm(...)]` attribute exists for destructive recipes if ever needed (not needed here).
- `set shell := ["bash", "-cu"]` is portable across macOS/Linux/CI Ubuntu runners. Windows is not a supported dev host (CLAUDE.md says nothing about Windows); document Windows-via-WSL in README if anyone asks.

### Pattern 7: Component tier — DiceUI column definition test (D-20)

**What:** A pure-logic test of a column `sortingFn`, `filterFn`, or `cell` render. Does not mount the full DiceUI data-table; tests the column def itself as a value. This is the "logic-heavy component" carve-out per D-03.

**Example sketch:** (Planner confirms exact file after reading `components/vehicles/vehicles-columns.tsx`.)
```tsx
// components/vehicles/vehicles-columns.test.tsx
import { describe, it, expect } from 'vitest';
import { render, screen } from '@testing-library/react';
import { vehiclesColumns } from './vehicles-columns';
// If vehiclesColumns exposes a filter/sort helper, test it directly.
// Otherwise render one cell in isolation inside a minimal wrapper.

describe('vehiclesColumns status filter', () => {
  it('filters rows by exact status match', () => {
    const statusCol = vehiclesColumns.find((c) => c.id === 'status' || (c as any).accessorKey === 'status');
```

### Anti-Patterns to Avoid

- **Importing Route Handlers directly in unit tests.** D-02 splits by concern: unit tier tests the `lib/` helpers; Route Handler files are exercised only over HTTP in the API tier. Direct imports also require mocking Next.js's request/response context, which is brittle.
- **Mounting shadcn primitive wrappers in component tests.** D-03 excludes them; they're covered transitively by E2E. Component tier is for logic-heavy units only.
- **Running Playwright in `just check`.** D-07 excludes Playwright tiers specifically to keep `just check` fast enough for pre-commit. Do not "optimize" by adding API tier to `check`.
- **Two Playwright configs.** Idiomatic pattern is one config, multiple projects (verified in ctx7 Playwright docs).
- **Relying on `vitest --watch` in `just`.** D-10 explicitly rejects `just test-watch`. Developers run `pnpm exec vitest` directly.
- **Hard-coding `http://localhost:3000`.** Use `PORT` env var + `baseURL` config; the `webServer` owns the port. Playwright's `page.goto('/vehicles')` resolves against `baseURL`.
- **Forgetting to add `tests/e2e/**` and `tests/api/**` to Vitest's `exclude`.** Otherwise Vitest will try to collect `*.spec.ts` files meant for Playwright.
- **Leaving `.auth/*.json` uncommitted but unignored.** Real session cookies would leak if a developer accidentally commits. `.gitignore` MUST contain `playwright/.auth/*.json` with a `!.gitkeep` exception.
- **Calling the reset endpoint from production.** The NODE_ENV gate is mandatory. Add a second env flag in Phase 1 (`TEST_RESET_ENABLED`) once real data is at stake.
- **Using `floatingPoint` arithmetic in `lib/money.ts`.** Re-stating from CLAUDE.md: the entire point of the D-19 example test is to lock the integer-cents rule. The test suite must contain an assertion that proves naive float math is not used (e.g., verify `add(10, 20) === 30` cents, not `0.3` dollars).

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| DOM environment for component tests | Manual jsdom setup | `environment: 'jsdom'` in `vitest.config.ts` | Vitest handles lifecycle, window reset, URL config out of the box |
| JSX/TSX transform for tests | Custom Babel/SWC wiring | `@vitejs/plugin-react` | Handles React 19 JSX runtime, Fast Refresh stripping, TSX parsing |
| Test webServer for Playwright | Custom `spawn('next', ['start'])` | `webServer` config block | Handles readiness-wait, reuseExistingServer, stdout plumbing, graceful shutdown |
| API test HTTP client | `axios`/`node-fetch` in specs | `playwright.request` fixture | Already parallelized per test, shares `baseURL` + `extraHTTPHeaders`, honors `storageState` for future authed requests |
| Auth state persistence between tests | Re-login per spec | `storageState` + `setup` project dependency | Playwright's documented pattern; one login produces a JSON file reused by every test |
| Reset-per-test plumbing | Global DB hooks per test | Dev-only HTTP endpoint (`POST /api/__test__/reset`) called in `beforeEach` | Keeps tests agnostic to storage (mock store today, Prisma tomorrow); stable contract across phases |
| Matchers (`toBeInTheDocument`, etc.) | Write assertions by hand | `@testing-library/jest-dom/vitest` | Standard DOM matchers, idiomatic failure messages |
| DOM event simulation | `fireEvent` with low-level `MouseEvent` | `@testing-library/user-event` | Simulates real user interaction with pointer/keyboard events in the right order |
| Coverage instrumentation | Istanbul manual wiring | `@vitest/coverage-v8` | Zero-config, uses V8's built-in coverage |
| Task runner with arg passing | Bash wrappers around pnpm scripts | `just` | Recipe dependencies, groups, default-help, optional positional params — all native |
| Integer currency parse/format | Hand-written `Number` tricks | Pure functions over integer cents only (no decimals anywhere except at the boundary) | Guarantees IEEE-754 never enters the calculation path |

**Key insight:** Every tool on this list is load-bearing against a known failure mode — `@vitejs/plugin-react` is not optional (SWC does not run in Vitest); `storageState` scaffolding now saves Phase 1 planning churn; the reset endpoint's stable contract is the single most important infrastructure decision because it absorbs the Phase 1 storage swap.

## Common Pitfalls

### Pitfall 1: Missing `@vitejs/plugin-react` — "Unexpected token" on `.tsx` imports
**What goes wrong:** First component test run fails with `SyntaxError: Unexpected token '<'` or `Failed to parse source for import analysis` when Vitest hits a `.tsx` file.
**Why it happens:** Vitest runs Vite, not Next.js's SWC transform. Without `@vitejs/plugin-react` registered in `vitest.config.ts` plugins, TSX never gets transformed.
**How to avoid:** Install the plugin and register it in `plugins: [react()]` from day one.
**Warning signs:** Any import of a `.tsx` file in a test file.

### Pitfall 2: Playwright collects Vitest files (or vice versa)
**What goes wrong:** `pnpm exec playwright test` tries to run `lib/money.test.ts`, or `pnpm exec vitest` tries to run `tests/e2e/*.spec.ts`.
**Why it happens:** Both runners default to globbing broad patterns.
**How to avoid:** Vitest config must set `exclude: [..., 'tests/e2e/**', 'tests/api/**']`. Playwright config must set `testDir: 'tests'` (not repo root) and per-project `testDir` for extra safety. Use `*.test.ts(x)` for Vitest, `*.spec.ts` for Playwright — the distinction is a convention worth locking.
**Warning signs:** Unexpected test counts, runner crashes inside files that clearly belong to the other tool.

### Pitfall 3: `webServer` timeout too low for cold builds
**What goes wrong:** CI fails with "Timed out waiting X ms from config.webServer" on the first run after checkout.
**Why it happens:** `next build && next start` can take 60–90s on a cold CI node. Playwright's default `webServer.timeout` is 60 seconds.
**How to avoid:** Set `timeout: 120_000` (2 minutes). D-12 already accepts ~30s; the budget above leaves headroom for CI.
**Warning signs:** Flaky first-run CI failures that pass on retry.

### Pitfall 4: Session cookies leak through `.auth/`
**What goes wrong:** A developer commits a real storageState file to git after Phase 1 lands. Real session cookies become public.
**Why it happens:** `.gitignore` wasn't scoped correctly, or only `.auth/` was ignored without `.gitkeep` exception, causing confusion.
**How to avoid:** Ignore `playwright/.auth/*.json`, commit `playwright/.auth/.gitkeep` so the dir exists, document in README.
**Warning signs:** Any JSON file under `playwright/.auth/` showing in `git status`.

### Pitfall 5: Reset endpoint shipped to production
**What goes wrong:** `POST /api/__test__/reset` hits production and wipes real data in Phase 1+.
**Why it happens:** NODE_ENV gate was forgotten, or the gate was correct but the handler still ran a truncation for non-production-but-not-dev environments (preview).
**How to avoid:** Belt-and-braces: `if (process.env.NODE_ENV === 'production') return 404;` plus a second explicit `TEST_RESET_ENABLED` flag that Phase 1 adds before swapping to Prisma truncation. Phase 0.3 only needs the NODE_ENV gate; document the second gate as a Phase 1 requirement.
**Warning signs:** The endpoint appears in the production Route Handler build output without a 404 branch.

### Pitfall 6: `@testing-library/jest-dom` imported at module top-level of test
**What goes wrong:** Matchers aren't registered, or import ordering shifts break setup.
**Why it happens:** Importing `@testing-library/jest-dom` directly rather than the `/vitest` subpath, or importing it in one test file instead of the setup file.
**How to avoid:** Import `'@testing-library/jest-dom/vitest'` once in `vitest.setup.ts`. Register the setup file via `setupFiles: ['./vitest.setup.ts']` in config.
**Warning signs:** `expect(...).toBeInTheDocument is not a function` errors.

### Pitfall 7: Floating-point creeping into `lib/money.ts`
**What goes wrong:** A developer writes `return cents / 100` during format and later writes `return a + b` over dollar values. CLAUDE.md non-negotiable violated.
**How to avoid:** The unit example test (D-19) must include at least one assertion that demonstrates integer-cents safety: e.g., `expect(add(10, 20)).toBe(30)` where `add` takes and returns cents only; `format(30)` returns the string `"$0.30"`. Add a second test that documents the trap: `expect(0.1 + 0.2).not.toBe(0.3)` — comment it as a pinned invariant.
**Warning signs:** Any `Math.round`, `toFixed`, or `Number(x).toString()` in `lib/money.ts`.

### Pitfall 8: RTL cleanup missing between tests
**What goes wrong:** Previous test's DOM leaks into the next test; queries return surprising results.
**Why it happens:** Vitest does not auto-cleanup RTL (unlike Jest with the `@testing-library/react` auto-cleanup hook under Jest).
**How to avoid:** Explicit `afterEach(() => cleanup())` in `vitest.setup.ts`, as shown in Pattern 1.
**Warning signs:** Tests that pass in isolation but fail in the full run; "found multiple elements" errors.

## Code Examples

Verified patterns from official sources:

### `lib/money.ts` minimal surface (D-19)
Pure-function API over integer cents. Planner owns the exact signature, but the surface stays this lean in Phase 0.3:

```ts
// lib/money.ts
// All values are integer cents. Boundary-only conversion.
export function add(a: number, b: number): number {
  return a + b;
}

export function subtract(a: number, b: number): number {
  return a - b;
}

// Display-only conversion; always produces a string, never a number.
export function format(cents: number, currency = 'USD'): string {
  const intl = new Intl.NumberFormat('en-US', { style: 'currency', currency });
  return intl.format(cents / 100);
}

// Parse user input ("1,234.56") into integer cents. Returns null on bad input.
export function parseInput(raw: string): number | null {
  const normalized = raw.replace(/[^\d.-]/g, '');
  if (!/^-?\d+(\.\d{1,2})?$/.test(normalized)) return null;
  const [whole, frac = '00'] = normalized.split('.');
  const cents = Number(whole) * 100 + Number(frac.padEnd(2, '0'));
  return Number.isFinite(cents) ? cents : null;
}
```
