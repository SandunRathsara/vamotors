# Phase 0.3: Testing Strategy - Context

**Gathered:** 2026-04-15
**Status:** Ready for planning

<domain>
## Phase Boundary

Stand up the three-tier test harness and `just` task runner so every future phase ships with tests. Deliverables:

1. `Justfile` at repo root with the agreed recipe surface
2. Vitest configured for unit + component tests (jsdom + @testing-library/react)
3. Playwright configured with two projects: E2E and API (both reusing one webServer)
4. One example test per tier, proving the harness works against the existing Phase 0.1 mock-data shell
5. `.auth/` fixture scaffolding ready for Phase 1 Better Auth integration

Writing the full test suite for each feature is NOT in this phase — each future phase owns its own tests. Phase 0.3 is the platform, not the content.

</domain>

<decisions>
## Implementation Decisions

### Test tiers and boundaries
- **D-01:** Three tiers: (a) unit/component via **vitest + jsdom + @testing-library/react**, (b) API via **@playwright/test** using `playwright.request`, (c) E2E via **@playwright/test** browser context.
- **D-02:** Route Handlers (`app/api/**`) are tested in **both** tiers with a split by concern: pure logic (Zod parsing, business rules, calculations) is extracted into `lib/` and unit-tested; the Route Handler file itself is exercised only through the API tier over real HTTP.
- **D-03:** Component tier covers **logic-heavy components only** — hooks, reducers, form validation, data-table column defs, utility components. shadcn primitive wrappers and purely presentational components are NOT unit-tested; they're covered transitively by E2E.
- **D-04:** Out of scope for this phase (and the suite overall until explicitly added): **perf/load testing**, **accessibility automation** (axe-core/a11y assertions), **visual regression** (screenshot diffing), **cross-browser matrix** — Chromium only for E2E.

### Just recipe surface
- **D-05:** `just test` runs all three tiers in sequence (unit → api → e2e).
- **D-06:** `just test <feature>` uses **path-prefix filtering** — runs any test file whose path contains the argument, across all tiers. No tag decorators required.
- **D-07:** `just check` is the fast pre-commit recipe and runs: **typecheck** (`tsc --noEmit`), **lint** (`eslint`), **format verify** (`prettier --check`), and **unit tier only** (vitest). Playwright tiers are excluded to keep `just check` fast.
- **D-08:** `just ci` is the single recipe CI invokes: `just check` + api tier + e2e tier + `pnpm build`. Mirrors the full gate locally.
- **D-09:** `just test-e2e-ui` opens Playwright UI mode for interactive E2E debugging.
- **D-10:** No `just test-watch` and no `just test-file` recipes — user explicitly did not request them. Developers can invoke `pnpm exec vitest` directly for watch.
- **D-11:** `just coverage` generates v8 coverage reports for the unit tier. No CI-enforced thresholds (see D-17).

### E2E and API infrastructure
- **D-12:** E2E tests run against a **production build** — Playwright `webServer` config does `next build && next start`. Cold start is acceptable (~30s) in exchange for trustworthy output. API tier reuses the same webServer.
- **D-13:** API tier uses **`playwright.request`** against the running server — same HTTP path that E2E uses. No direct handler imports at this tier (direct imports belong to the unit tier per D-02).
- **D-14:** **Reset-per-test** via a dev-only endpoint: Phase 0.3 adds `POST /api/__test__/reset` (guarded to non-production NODE_ENV) that rehydrates the in-memory mock-data store. Tests call it in `beforeEach`. Future phases swap the implementation to truncate Prisma tables — the endpoint contract stays stable.
- **D-15:** **Auth fixturing via Playwright `storageState` files**, one per role. Phase 0.3 scaffolds the `globalSetup` pattern + `.auth/` directory structure and adds `.auth/` to `.gitignore`. Actual login flow is wired in Phase 1 when Better Auth lands — the scaffolding is ready so Phase 1 only adds the login helper.

### File layout
- **D-16:** Unit + component test files are **colocated** as `*.test.ts(x)` next to source. Example: `lib/money.ts` + `lib/money.test.ts`. Matches vitest defaults and survives refactors.
- **D-17:** E2E and API tests live in **`tests/e2e/`** and **`tests/api/`** respectively. Top-level, split by tier.

### Coverage
- **D-18:** **v8 coverage, no thresholds**. `just coverage` produces HTML + text reports for inspection. No CI gate — early-stage project, thresholds create noise before patterns settle. Revisit in a later phase.

### Example tests to ship (prove each tier)
- **D-19:** **Unit example** — integer-cents money/currency helper (`lib/money.ts` or equivalent). Pure-function test, demonstrates the unit tier and directly supports the CLAUDE.md non-negotiable of integer currency storage.
- **D-20:** **Component example** — one Phase 0.2 data-table column definition test (sort/filter logic). Demonstrates vitest + RTL.
- **D-21:** **API example** — `playwright.request` hits one existing Phase 0.1 mock Route Handler, asserts response shape + status.
- **D-22:** **E2E example** — one golden-path navigation flow against mock data (e.g., dashboard → vehicles list → vehicle detail). Proves `webServer`, fixtures, and browser context all wire together.

### Claude's Discretion
- Exact vitest config shape (`vitest.config.ts` vs merged with Next.js), jsdom environment setup, setup files.
- Exact Playwright config layout (single `playwright.config.ts` with two projects vs two configs).
- Justfile style conventions (recipe grouping, comments, help output).
- Naming/location of the test-reset endpoint guard mechanism.
- Whether to install `@vitejs/plugin-react` vs rely on SWC.
- Package manager — **pnpm** per CLAUDE.md (despite `bun.lock` being present in the repo).

</decisions>

- **pnpm** is the package manager (per CLAUDE.md) — installs and scripts must use `pnpm`, not bun, despite the stray `bun.lock`.

- **No existing test tooling** — clean slate, no migration concerns.
- **ESLint + Prettier already configured** — `just check` wires to existing `pnpm lint` / prettier setup, does not reinstall.

- `app/api/__test__/reset/route.ts` — new dev-only endpoint (D-14).

<specifics>
## Specific Ideas

- CI wiring is **deferred** — `just ci` is the contract; a later phase adds GitHub Actions wiring that calls it. Phase 0.3 does not touch `.github/workflows/`.
- The test-reset endpoint (`POST /api/__test__/reset`) is designed to **survive Phase 1+** — the contract stays stable while the implementation switches from "rehydrate in-memory store" to "truncate Prisma tables".
- Auth fixturing is **scaffolded but empty** in Phase 0.3. The `.auth/` directory, `globalSetup` hook, and per-role storageState pattern all exist; Phase 1 plugs in the actual login calls.
- `just check` is intended to be fast enough to run on every save / pre-commit — that's why the Playwright tiers are excluded.

</specifics>

<deferred>
## Deferred Ideas

- **GitHub Actions / CI pipeline wiring** — deferred to a dedicated CI/devops phase. `just ci` is ready whenever that phase lands.
- **Visual regression testing** (Chromatic/Percy/Playwright screenshot diffing) — out of scope, revisit if visual drift becomes a real problem.
- **Accessibility automation** (axe-core, jest-axe) — out of scope. Manual a11y review during UI phases stays the bar.
- **Performance / load testing** (k6, Lighthouse CI) — out of scope.
- **Cross-browser matrix** (Firefox, WebKit) — out of scope; Chromium only.
- **Coverage thresholds / enforcement** — deferred until patterns settle. `just coverage` reports exist, no gate.
- **Watch-mode `just` recipe** and **single-file `just` recipe** — user explicitly skipped; re-add if developer friction shows up.
- **Mutation testing**, **contract testing**, **property-based testing** — none raised; noted here only so future phases don't treat their absence as an oversight.

</deferred>
