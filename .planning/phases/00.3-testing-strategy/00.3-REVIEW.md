# Phase 00.3: Code Review Report

**Reviewed:** 2026-04-15
**Depth:** standard
**Files Reviewed:** 14
**Status:** issues_found

## Summary

Phase 00.3 ships a three-tier test harness (Vitest unit/component + Playwright API + Playwright E2E) and four example tests. Overall quality is high: `lib/money.ts` correctly implements integer-cents arithmetic per the CLAUDE.md float-safety rule, the test examples are well-structured, and the `/api/test/reset` security gate is sound in principle.

The most important finding is **WR-01**: the `TEST_RESET_ENABLED` env-var gate on `/api/test/reset` is a single-signal override that — if accidentally set in a Vercel production environment — fully opens a destructive endpoint. Today the blast radius is a wiped in-memory mock, but the existing comment in `lib/mock-store.ts:172-176` explicitly promises the handler's body will swap to `prisma.$transaction([...deleteMany])` in Phase 1. Please harden this gate (or add a hard assertion against `VERCEL_ENV === "production"`) before that swap lands.

No Critical issues. Four Warnings, five Info items.

## Warnings

### WR-01: Destructive test endpoint gated by a single env var with no defense-in-depth

**File:** `app/api/test/reset/route.ts:19-23`
**Issue:** `TEST_RESET_ENABLED=1` is sufficient on its own to open the gate in any environment. A single typo in a Vercel dashboard env setting, or a leaked `.env` merged into production, turns this into a public unauthenticated destructive endpoint. Today it resets an in-memory mock; Phase 1 (per the comment in `lib/mock-store.ts:172-176`) replaces the body with `prisma.$transaction([...deleteMany])`, at which point an accidental env flip becomes a data-loss event.
**Fix:** Add a second independent signal so no single env var can satisfy the gate. Minimum viable hardening — assert against `VERCEL_ENV` so production deploys can never open the gate:

```ts
export async function POST(request: Request) {
  const isProd = process.env.NODE_ENV === "production";
  const isVercelProd = process.env.VERCEL_ENV === "production";
  const testOverride = process.env.TEST_RESET_ENABLED === "1";

  // Vercel production can NEVER open the gate, regardless of other env vars.
  if (isVercelProd) {
    return new NextResponse("Not Found", { status: 404 });
  }
  if (isProd && !testOverride) {
    return new NextResponse("Not Found", { status: 404 });
  }

  // Optional but recommended: signed header check
  const secret = process.env.TEST_RESET_SECRET;
  const headerSecret = request.headers.get("x-test-reset-secret");
  if (secret && headerSecret !== secret) {
    return new NextResponse("Not Found", { status: 404 });
  }

  resetAll();
  return NextResponse.json({ ok: true });
}
```

Playwright then sets both `TEST_RESET_ENABLED=1` and `TEST_RESET_SECRET=...` in `playwright.config.ts` and passes the header via `extraHTTPHeaders`.

### WR-02: Reset handler ignores the request; no origin/auth check and no `SECURITY TODO` breadcrumb

**File:** `app/api/test/reset/route.ts:18`
**Issue:** The handler does not read the `Request` at all — no origin check, no body, no auth header. For a dev-mode endpoint this is acceptable, but the inline comment promises Phase 1 hardening and there is no `TODO` breadcrumb in the code to make sure that promise is kept.
**Fix:** Add an explicit comment listing the hardening items so Phase 1 reviewers see them:

```ts
// SECURITY TODO(phase-1): before swapping resetAll() to prisma.$transaction,
// add: (1) signed header check, (2) origin allow-list, (3) rate limit,
// (4) VERCEL_ENV !== "production" hard assert. See WR-01 in 00.3-REVIEW.md.
export async function POST(request: Request) { ... }
```

### WR-03: `tests/auth.setup.ts` navigates to `/` which may not exist, risking cascading setup failure

**File:** `tests/auth.setup.ts:12`
**Issue:** The setup project calls `page.goto('/')` purely because `context.storageState()` conventionally runs after a navigation. If `/` does not exist in the current Phase 0.1 app (the golden-path E2E in `tests/e2e/dashboard.spec.ts` goes directly to `/dashboard`), or if it redirects to an auth page that hangs Playwright's navigation wait, the setup project fails and takes both the `api` and `e2e` projects with it (both declare `dependencies: ['setup']`).
**Fix:** Either target a known-good route, or drop the navigation entirely — `storageState()` can be written from a fresh context without visiting any page:

```ts
setup('scaffold empty auth state', async ({ browser }) => {
  await mkdir(dirname(userFile), { recursive: true });
  const context = await browser.newContext();
  await context.storageState({ path: userFile });
  await context.close();
});
```

### WR-04: `toCents` can produce `-0` for inputs like `"-0"` or `"-0.00"`

**File:** `lib/money.ts:17-25`
**Issue:** The regex correctly rejects malformed inputs like `"-.5"` and `"1."`, but `toCents("-0")` evaluates to `-1 * (0 * 100 + 0)` === `-0`. While `-0 === 0` is `true`, `Object.is(-0, 0)` is `false` and `JSON.stringify(-0)` returns `"0"` — this asymmetry can leak into equality checks, sorted outputs, and fixture diffs in surprising ways.
**Fix:** Normalize the zero case:

```ts
const result = sign * (parseInt(wholeAbs, 10) * 100 + parseInt(paddedFrac, 10));
return result === 0 ? 0 : result; // normalize -0 → 0
```

Add a regression test: `expect(Object.is(toCents("-0"), 0)).toBe(true);`

## Info

### IN-01: Coverage include list omits `app/` directory

**File:** `vitest.config.ts:26`
**Issue:** `include: ['lib/**', 'components/**', 'hooks/**']` means API route handlers (e.g. `app/api/test/reset/route.ts`) will never appear in Vitest coverage. Logic extracted from route handlers into unit-testable helpers will still be covered via `lib/`, but the routes themselves will show 0% even when exercised by the Playwright `api` tier.
**Fix:** Either add `'app/**'` to the include list, or document in the config comment that API route coverage is delegated to the Playwright `api` tier.

### IN-02: `MockStore.query` sort is stable-by-spec but treats `undefined` and `""` identically

**File:** `lib/mock-store.ts:101-113`
**Issue:** `Array.prototype.sort` is stable per ES2019+, so this is not broken in modern Node, but `String(aVal ?? "")` groups `undefined` with empty strings which can make pagination results order-dependent on insertion order when a filter key is partially populated.
**Fix:** Add a deterministic secondary sort by `id`, or add a comment documenting the convention.

### IN-03: `MockStore.query` full-text search is shallow

**File:** `lib/mock-store.ts:79-86`
**Issue:** The search only scans top-level string fields. Nested objects (e.g. a `customer.name` on a `Sale`) won't match. Probably intentional for the mock layer, but the Phase 1 Prisma swap reviewer will need to know the intended semantics.
**Fix:** Add a comment: `// Shallow search by design — matches Prisma WHERE OR on top-level string columns only.`

### IN-04: API spec `"is reachable"` test double-resets

**File:** `tests/api/vehicles.api.spec.ts:38-45`
**Issue:** The file-level `test.beforeEach` posts to `/api/test/reset`, then the `"is reachable in dev/test mode"` test posts again. Harmless but the intent (prove the endpoint returns 200) is duplicated by the beforeEach itself. If the beforeEach ever fails, this test also fails — the smoke-test assertion adds no independent signal.
**Fix:** Either delete this test (the beforeEach already proves reachability) or scope it under a nested `describe` that overrides `beforeEach` to a no-op so the smoke test is a standalone probe.

### IN-05: `vehicles-columns.test.tsx` uses unsafe type casts instead of TanStack's `ColumnDef` discriminated union

**File:** `components/vehicles/vehicles-columns.test.tsx:29,38,44,50`
**Issue:** Multiple `as { accessorFn?: ... }` / `as { cell?: ... }` / `as { meta?: ... }` casts lose type safety. If the column shape changes, these casts silently degrade to `undefined` checks rather than failing at compile time. TanStack Table exposes `ColumnDef<TData>` with typed `AccessorFnColumnDef` and `DisplayColumnDef` variants that can be narrowed via `'accessorFn' in col`.
**Fix:**

```ts
import type { ColumnDef } from "@tanstack/react-table";

const cols = vehicleColumns as ColumnDef<Vehicle>[];
const col = cols.find((c) => c.id === "makeModel");
if (!col || !('accessorFn' in col)) {
  throw new Error("makeModel column missing accessorFn");
}
expect(col.accessorFn!({ make: "Toyota", model: "Corolla" } as Vehicle, 0))
  .toBe("Toyota Corolla");
```
