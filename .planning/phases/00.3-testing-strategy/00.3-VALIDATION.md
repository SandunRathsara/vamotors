# Phase 0.3 — Validation Strategy

> Per-phase validation contract for feedback sampling during execution.
> Source of truth: `00.3-RESEARCH.md` § Validation Architecture (maps D-01..D-22 to verification mechanisms).

---

## Test Infrastructure

| Property | Value |
|----------|-------|
| **Framework** | Vitest 4.1.4 (unit/component) + @playwright/test 1.59.1 (api + e2e) |
| **Config file** | `vitest.config.ts`, `playwright.config.ts` (both installed Wave 0) |
| **Quick run command** | `just check` (typecheck + lint + format + vitest unit tier) |
| **Full suite command** | `just ci` (check + api + e2e + pnpm build) |
| **Estimated runtime** | ~5s quick / ~60–120s full (Playwright cold-start `next build && next start`) |

---

## Sampling Rate

- **After every task commit:** Run `just check` (unit tier only — Playwright excluded to stay fast per D-07)
- **After every plan wave:** Run `just ci` (all three tiers + build)
- **Before `/gsd-verify-work`:** Full `just ci` must be green
- **Max feedback latency:** 10 seconds for `just check`, 150 seconds for `just ci`

## Per-Decision Validation Map (D-01..D-22)

> Mirrors `00.3-RESEARCH.md` § Validation Architecture. Each CONTEXT.md decision has an explicit verification mechanism. gsd-planner uses this to assign `<automated>` checks in PLAN.md tasks.

| Decision | What it locks | How it's validated |
|----------|---------------|--------------------|
| D-01 | Three tiers (vitest/jsdom/RTL + playwright request + playwright e2e) | `just test` runs all three; each tier produces ≥1 passing example |
| D-02 | Route Handler split (pure logic in `lib/` → unit; HTTP path → API tier) | D-19 unit test lives in `lib/`; D-21 API test hits HTTP endpoint only |
| D-03 | Component tier covers logic-heavy only | D-20 targets DiceUI column def helper (logic), not a shadcn primitive |
| D-04 | No perf/a11y/visual/cross-browser | Playwright projects contain only Chromium; no axe-core/lighthouse/screenshot deps |
| D-05 | `just test` runs unit → api → e2e sequentially | Justfile recipe body; `just test --dry-run` inspection |
| D-06 | `just test <arg>` path-prefix filter | Justfile recipe uses positional arg piped to vitest + playwright `--grep` or path filter |
| D-07 | `just check` = typecheck + lint + format + unit only | Recipe body contains `tsc --noEmit`, `pnpm lint`, `prettier --check`, `pnpm test:unit`; NO playwright calls |
| D-08 | `just ci` mirrors full gate locally | Recipe body chains check + api + e2e + `pnpm build` |
| D-09 | `just test-e2e-ui` opens Playwright UI | Recipe runs `pnpm exec playwright test --ui` |
| D-10 | No watch/single-file recipes | Justfile grep confirms absence |
| D-11 | `just coverage` → v8 unit coverage | Recipe runs `pnpm exec vitest run --coverage`; `vitest.config.ts` `coverage.provider: 'v8'` |
| D-12 | Playwright `webServer` uses prod build | `playwright.config.ts` `webServer.command === 'pnpm build && pnpm start'` |
| D-13 | API tier uses `playwright.request` | `tests/api/*.spec.ts` uses `request` fixture; grep confirms no direct route imports |
| D-14 | `POST /api/__test__/reset` dev-only, rehydrates mock store | Endpoint returns 404 when `NODE_ENV=production`; E2E + API `beforeEach` POST succeeds in dev/test |
| D-15 | Playwright `storageState` scaffolded, empty in 0.3 | `tests/auth.setup.ts` exists as setup project; `.auth/` gitignored |
| D-16 | Unit/component tests colocated `*.test.ts(x)` | `lib/money.test.ts` + `components/**/*.test.tsx` next to source |
| D-17 | E2E in `tests/e2e/`, API in `tests/api/` | Playwright `projects[].testDir` paths |
| D-18 | v8 coverage, no thresholds | Config has coverage provider but NO `coverage.thresholds` entries |
| D-19 | Unit example — integer-cents money helper | `lib/money.test.ts` passes; includes invariant assertion `expect(0.1 + 0.2).not.toBe(0.3)` |
| D-20 | Component example — DiceUI column def | `components/**/*-columns.test.tsx` passes (targets `vehicles-columns.tsx` per research Q1) |
| D-21 | API example — playwright.request against Phase 0.1 mock route | `tests/api/*.spec.ts` asserts response shape + status via `request.get()` |
| D-22 | E2E example — golden-path navigation | `tests/e2e/*.spec.ts` traverses dashboard → vehicles list → vehicle detail |
