# Phase 0.1: Frontend Shell Migration — Context

**Gathered:** 2026-04-11
**Status:** Ready for planning

<domain>
## Phase Boundary

A clickable Next.js 15 App Router frontend shell covering all 34 reviewed wireframes in `design/*.html`, built on the `radix-vega` shadcn preset with mock data served through Next.js Route Handlers. Pure UI layer — no auth (Better Auth lands in Phase 1), no database (Prisma/Neon land in Phase 1), no business logic, no file uploads to R2 (real R2 integration lands in Phases 2–3). The deliverable is what a client walks through on a laptop during an in-person sign-off before backend feature work begins.

**What "clickable" means here:** every sidebar and topbar link reaches a real route, every list screen paginates/sorts/filters against mock Route Handlers, every form posts to a mock handler that mutates an in-memory store, every mutation shows a sonner toast, no 404s on the reviewed screen list, and error/loading states are visible instead of blank flashes.

**What this phase is NOT:** a real authentication flow, a real database, a real R2 uploader, real business rules (advance expiry tiers, cost basis, status machine enforcement), or a production deployment. All of those ship in Phases 1–10.

> **⚠ ROADMAP alignment note:** Success criterion #3 in ROADMAP.md currently reads *"renders with Stitch-derived design tokens."* The design-system decision locked in this context (D-06) drops the Stitch palette and uses the `radix-vega` preset instead. ROADMAP success criterion #3 needs to be rewritten to reference the radix-vega preset tokens before planning is approved. The `gsd-planner` agent should surface this as a plan-check flag.

</domain>

## Implementation Decisions

### Techstack — core toolchain

- **D-01:** Scaffold with `pnpm dlx shadcn@latest init --preset b2BoX602a --template next`. The preset has already been verified to produce: Next.js `16.1.7`, React `19.2.4`, Tailwind `v4.2.1`, shadcn CLI `^4.2.0`, Lucide icons, `tw-animate-css`, `next-themes`, `radix-ui`, a `zinc` base + green primary palette, and native `--sidebar-*` / `--chart-*` tokens. Do **not** run `create-next-app` manually; the preset owns scaffolding.
- **D-02:** After scaffolding, pin every dependency to an exact version in `package.json` (strip all `^` and `~`). The preset ships caret ranges; they must be rewritten before the first commit. Same for devDependencies.
- **D-03:** Accept the preset's Next.js `16.1.7` unless the post-scaffold pin step detects a breaking difference with `16.2.2` (documented latest). If a bump is warranted, do it as a single dedicated commit with reasoning. Do **not** mix scaffold and bump in the same commit.
- **D-04:** Node floor: whatever `next@16.1.7` + `react@19.2.4` require (document the resolved version in README; do not invent a floor). Package manager: pnpm (already locked by PROJECT.md). Turbopack dev is enabled by the preset — keep it.

### Techstack — TanStack and data libraries

- **D-05:** Install **TanStack Table v8**, **TanStack Form v1**, and **TanStack Query v5**. Wire up `QueryClient` in a client `Providers` component. Do **not** install TanStack Virtual (fixtures are small, premature) and do **not** install TanStack Router (Next.js App Router owns routing).

### Techstack — shadcn registries and third-party component libraries

- **D-06:** Enable four component registries:
  1. `@shadcn` — core primitives + official Blocks (dashboard, sidebar, login, data-table, chart)
  2. `@originui` — richer form inputs (phone, tag, multi-select, combobox variants, OTP, date range pickers)
  3. `@diceui` — **the data table** (`@diceui/data-table` built on `@tanstack/react-table` + `nuqs` + shadcn primitives) is the authoritative list-screen component for Phase 0.1 and beyond. Also evaluate @diceui Kanban, Sortable, Combobox, Tags-input, Mention, Tree, Masonry per-screen.
  4. `@magicui` — available if any animation flourish is wanted, but use sparingly (B2B dealership ops tool, not a marketing site).
- **D-07:** Supporting libraries installed alongside the preset: `sonner` (toasts), `nuqs` (URL state for table filters/pagination — required by @diceui data-table), `react-day-picker` via shadcn Calendar/DatePicker, `cmdk` via shadcn Command, `recharts` via shadcn Chart (dashboard KPIs + report visuals), `embla-carousel-react` via shadcn Carousel, `vaul` via shadcn Drawer (mobile sidebar sheet).

### Visual Design System

- **D-08:** **Drop the Stitch Material 3 palette entirely.** The `design/*.html` wireframes are **information-architecture references only** (what screens exist, what data each shows, what columns/filters/sections they have, how navigation links together). They are **not** visual references. Colors, typography, icons, spacing, radius, shadows, component chrome — all come from the `radix-vega` preset.
- **D-09:** Ship dark mode from day one. `next-themes` with `defaultTheme="system"` + a toggle in the topbar. The preset already includes the full `.dark` token block.
- **D-10:** Typography: Inter via `next/font/google` with `display: "swap"` and `variable: "--font-sans"`, wired into the preset's `--font-sans` slot. Do **not** use Google Fonts CDN at runtime — self-host through next/font.
- **D-11:** Icons: Lucide React (preset default). Do **not** re-introduce Material Symbols. When a wireframe uses a Material Symbol with no clear Lucide equivalent, pick the closest match; do not ship both icon systems.
- **D-12:** Accept the preset's `--radius: 0.625rem` and derived sm/md/lg/xl/2xl/3xl/4xl radius scale. Accept the preset's green/teal primary hue as the client's approved brand color for Phase 0.1 sign-off.

### Component Strategy

- **D-13:** **Blocks-first, custom where needed.** For any screen where an official shadcn Block is close (dashboard-01 → `dashboard.html`, sidebar-07 → the 17-link nav, login-04 → `login.html`, data-table blocks → list screens), start from the Block and adapt. For screens without a matching block (vehicle-detail with tabs, purchase wizard, lease-deal-detail with its tabbed workflow), compose from shadcn primitives + @diceui + @originui. No hand-rolled markup.
- **D-14:** Route tree uses a `(app)` route group with a single shared layout that owns the sidebar + topbar + theme provider. Routes:
  - `/login` (outside the group, its own minimal layout)
  - `/(app)/dashboard`
  - `/(app)/vehicles`, `/(app)/vehicles/[id]`
  - `/(app)/purchases`
  - `/(app)/sales`, `/(app)/sales/[id]`
  - `/(app)/repairs`
  - `/(app)/customers`, `/(app)/customers/[id]`
  - `/(app)/third-parties`, `/(app)/third-parties/[id]` (dynamic — resolves supplier / garage / finance variants from the fixture `type` field; the three existing detail wireframes become render branches, not separate routes)
  - `/(app)/approvals`
  - `/(app)/reports` (hub)
  - `/(app)/reports/stock`, `/reports/sales`, `/reports/purchases`, `/reports/repairs`, `/reports/advances`, `/reports/profit`, `/reports/audit` (static routes — each report has a unique column/filter set, so `[slug]` would save nothing)
  - `/(app)/cash-flow`
  - `/(app)/lease-deals`, `/(app)/lease-deals/[id]`
  - `/(app)/lease-comparison`
  - `/(app)/lease-rate-sheets`
  - `/(app)/lease-dispatch`
  - `/(app)/lease-reconciliation`
  - `/(app)/settings`
  - `/(app)/users`
- **D-15:** The shared shell is built with shadcn v4's `<Sidebar>` component (native `--sidebar-*` tokens from the preset). The 17 links are grouped into five `<SidebarGroup>` sections — **Main** (Dashboard, Vehicles, Purchases, Sales, Repairs), **Management** (Customers, Third Parties, Reports, Approvals), **Finance** (Reconciliation, Cash Flow), **Brokerage** (Lease Deals, Comparison Tool, Rate Sheets, Dispatch), **System** (Settings, Users & Roles). Approvals item shows a mock unread badge sourced from `/api/approvals?status=pending`. Mobile collapses via Vaul (shadcn Drawer / Sheet). Keyboard shortcuts from `<SidebarProvider>` left on.
- **D-16:** Topbar is a separate `<Header>` sibling above `<SidebarInset>` inside `/(app)/layout.tsx`. It contains: global search input bound to `cmdk` Command palette (Ctrl+K / Cmd+K), notification bell popover reading from `/api/notifications`, help popover linking to the docs under `docs/`, avatar dropdown with Profile / Settings (→ `/settings`) / Logout (→ `/login`). Every control leads somewhere — no dead hrefs.

### Mock Data Architecture

- **D-17:** Mock data flows through **Next.js Route Handlers** under `/api/*`, not through direct fixture imports. Client components fetch via TanStack Query with stable query keys. This exercises the same request/response shapes Phase 1 backends will implement and prevents contract drift.
- **D-18:** Handlers accept real pagination, sorting, and filtering params (`page`, `pageSize`, `sortBy`, `sortDir`, `q`, `status`, `dateFrom`, `dateTo`) and apply them in-memory against the fixtures. Return shape: `{ data: T[], total: number, page: number, pageSize: number }`. Single-record GETs return `notFound()` for unknown ids. Mutations return the mutated record.
- **D-19:** Handlers inject a **random 200–600ms latency** per request so skeletons, loading states, and sonner toasts are visible during the sign-off demo. Configurable via an env flag (`MOCK_LATENCY_MS_MIN` / `MOCK_LATENCY_MS_MAX`) so the exec can disable it if the client asks.
- **D-20:** Fixture scale is **realistic-medium**: ~150 vehicles across all statuses, ~80 customers, ~100 sales across all four sale types, ~40 repairs, ~30 approval requests, ~20 lease deals, ~30 third parties (split across supplier / garage / finance types). Fixtures live under `lib/mock-data/*.ts` with shared Zod schemas in `lib/mock-data/schemas.ts`. Fixtures are **hand-authored**, not generated with faker.js, so the client sees realistic dealership data (real-looking VINs, model names, prices) during sign-off.
- **D-21:** Mutations use an **in-memory store** scoped to the Node process. POST/PATCH/DELETE handlers mutate a module-scoped Map keyed by id. Lifetime = until `pnpm dev` restarts. This is a feature, not a bug: every fresh dev run gives a clean demo environment. Do **not** persist to localStorage or disk — it creates cache-invalidation ambiguity and blurs the source of truth.
- **D-22:** **Every successful mutation fires a sonner toast**, every failed mutation fires an error toast. This is how the client sees that clicks are doing something.

### Wireframe Migration Scope

- **D-23:** **All 34 wireframes are migrated in Phase 0.1.** Partial scope contradicts ROADMAP success criterion #3 ("every reviewed wireframe has a corresponding route"). The client needs full coverage to evaluate phases 1–10, especially the 6 lease brokerage screens (phases 8–9) and 8 report screens (phase 7) that only this phase will show them before their respective backend phases.
- **D-24:** The distinction between the 12 Stitch-updated wireframes and the 22 non-Stitch ones is **irrelevant** now that we're not using Stitch tokens. All 34 are IA sources for layout and content only. No pre-migration re-design pass is needed.
- **D-25:** **Sign-off is in-person via `pnpm dev`, not a Vercel preview deploy.** README must document the local setup clearly: prerequisites (Node version, pnpm version), install steps, dev command, where the mock Route Handlers live, how to reset the in-memory store. No Vercel account setup in Phase 0.1.
- **D-26:** **File uploads (vehicle photos, DO documents) use a local dropzone with `URL.createObjectURL` previews.** No R2, no presigned URLs, no network. Drag files onto vehicle-detail / lease-deal-detail / sale-detail and they render in the on-page gallery. Metadata (filename, size, count) lives in the in-memory store with the parent record. When Phase 2 swaps in real presigned PUT, only the upload adapter changes — the UI component contract stays the same.
- **D-27:** **Login page is static.** Any email/password combo clicks "Sign In" and routes to `/dashboard`. No auth, no validation (beyond the form's required-field hint). `/(app)` routes are **not** guarded — navigable directly. Phase 1 replaces this with Better Auth credential sign-in.
- **D-28:** **Error handling:** custom `app/not-found.tsx` (branded, links back to dashboard), `app/error.tsx`, and a `loading.tsx` per route segment so the 200–600ms API delay renders a shimmer instead of a blank page. Detail pages (`[id]`) call `notFound()` for unknown ids. Route Handlers return real HTTP error shapes (400 for bad params, 404 for missing records) so Query hooks resolve their error branches.
