## Summary

Phase 0.1 scaffolds a greenfield Next.js 16 App Router project using the `radix-vega` shadcn preset (preset code `b2BoX602a`), then migrates all 32 HTML wireframes in `design/` into live Next.js routes with mock Route Handler data. The deliverable is a clickable, dark-mode-capable shell the client reviews in-person before any backend work begins.

The stack is well-understood and fully verified. The preset scaffolds Next.js 16.1.7, React 19.2.4, Tailwind v4.2.1, and shadcn CLI v4.2.0 in a single command — no manual `create-next-app`. The `@diceui/data-table` component is the authoritative list-screen primitive, installed via the shadcn CLI from `diceui.com/r/data-table` and requiring `nuqs` for URL state. TanStack Query v5 wires the client layer to mock Route Handlers. All mock mutations land in a module-scoped in-memory Map with 200–600ms artificial latency.

One discrepancy surfaced: the CONTEXT.md references "34 wireframes" but the actual `design/*.html` count is **32 files**. The route count in D-14 also matches 32 screens (the three third-party detail files collapse to one dynamic route). The planner should treat 32 as the canonical wireframe count and update any success criterion language that says "34." The ROADMAP success criterion #3 also needs updating from "Stitch-derived design tokens" to "radix-vega preset tokens" per the CONTEXT.md flag.

**Primary recommendation:** Execute the scaffold (D-01), immediately pin all versions (D-02), install @diceui/data-table + supporting libraries, wrap the root layout with `NuqsAdapter` + `ThemeProvider` + `QueryClientProvider`, then implement screens wave-by-wave: shell first, then list screens (data tables), then detail screens (tabs + forms), then report screens.

### Pattern 1: Root Layout with ThemeProvider + Providers

```tsx
// app/layout.tsx (root)
// Source: Context7 /pacocoursey/next-themes + /tanstack/query
import { ThemeProvider } from 'next-themes'
import { Inter } from 'next/font/google'

const inter = Inter({ subsets: ['latin'], variable: '--font-sans', display: 'swap' })

export default function RootLayout({ children }: { children: React.ReactNode }) {
  return (
    <html lang="en" suppressHydrationWarning>
      <body className={inter.variable}>
        <ThemeProvider attribute="class" defaultTheme="system" enableSystem>
          <Providers>{children}</Providers>
        </ThemeProvider>
      </body>
    </html>
  )
}
```

```tsx
// components/layout/providers.tsx
'use client'
// Source: Context7 /tanstack/query + /47ng/nuqs
import { QueryClient, QueryClientProvider, isServer } from '@tanstack/react-query'
import { NuqsAdapter } from 'nuqs/adapters/next/app'

let browserQueryClient: QueryClient | undefined

function getQueryClient() {
  if (isServer) return new QueryClient({ defaultOptions: { queries: { staleTime: 60_000 } } })
  if (!browserQueryClient) browserQueryClient = new QueryClient({ defaultOptions: { queries: { staleTime: 60_000 } } })
  return browserQueryClient
}

export function Providers({ children }: { children: React.ReactNode }) {
  const queryClient = getQueryClient()
  return (
    <QueryClientProvider client={queryClient}>
      <NuqsAdapter>{children}</NuqsAdapter>
    </QueryClientProvider>
  )
}
```

### Pattern 3: Mock Route Handler with Latency + Pagination

```typescript
// app/api/vehicles/route.ts
// Source: Context7 /vercel/next.js/v16.2.2 Route Handler docs
import { type NextRequest, NextResponse } from 'next/server'
import { vehiclesStore } from '@/lib/mock-store'
import { injectLatency } from '@/lib/utils'

export async function GET(request: NextRequest) {
  await injectLatency()
  const { searchParams } = request.nextUrl
  const page = parseInt(searchParams.get('page') ?? '1', 10)
  const pageSize = parseInt(searchParams.get('pageSize') ?? '20', 10)
  const q = searchParams.get('q') ?? ''
  const status = searchParams.get('status') ?? ''
  const sortBy = searchParams.get('sortBy') ?? 'purchaseDate'
  const sortDir = searchParams.get('sortDir') ?? 'desc'

  let data = vehiclesStore.getAll()
  if (q) data = data.filter(v => `${v.make} ${v.model}`.toLowerCase().includes(q.toLowerCase()))
  if (status) data = data.filter(v => v.status === status)
  // apply sorting...
  const total = data.length
  const paginated = data.slice((page - 1) * pageSize, page * pageSize)
  return NextResponse.json({ data: paginated, total, page, pageSize })
}

// lib/utils.ts — latency helper
export async function injectLatency() {
  const min = parseInt(process.env.MOCK_LATENCY_MS_MIN ?? '200', 10)
  const max = parseInt(process.env.MOCK_LATENCY_MS_MAX ?? '600', 10)
  const ms = min + Math.random() * (max - min)
  return new Promise(resolve => setTimeout(resolve, ms))
}
```

- **Do NOT destructure params without awaiting:** `{ id } = params` fails in Next.js 15+. Always `const { id } = await params`.
- **Do NOT use `PDFDownloadLink` from @react-pdf/renderer** in Phase 0.1 — deferred to Phase 5. Use a disabled button stub.

- **Do NOT use `tailwindcss-animate` plugin** — use `tw-animate-css` (imported in globals.css, already in preset).

## Common Pitfalls

### Pitfall 1: @diceui Import Paths Need Manual Fix After Install

**What goes wrong:** After `pnpm dlx shadcn@latest add "https://diceui.com/r/data-table"`, the CLI installs files but does not correctly resolve custom component paths in three files.
**Why it happens:** shadcn CLI doesn't handle the diceui component's internal import aliases correctly.
**How to avoid:** Immediately after install, manually update imports in:
1. `components/data-table/data-table.tsx` — fix `getCommonPinningStyles` import to point to `@/lib/data-table`
2. `lib/data-table.ts` — fix `dataTableConfig` + `ExtendedColumnFilter`/`FilterOperator`/`FilterVariant` types to `@/config/data-table` and `@/types/data-table`
3. `lib/parsers.ts` — relocate `dataTableConfig` and column types to dedicated paths
**Warning signs:** TypeScript errors about missing exports immediately after install.
[CITED: diceui.com/docs/components/radix/data-table]

### Pitfall 2: NuqsAdapter Missing from Root Layout

**What goes wrong:** @diceui/data-table uses nuqs internally. Without `NuqsAdapter` wrapping the layout, table URL state throws a runtime error.
**Why it happens:** nuqs requires an adapter to know which router implementation to use.
**How to avoid:** Add `<NuqsAdapter>` in the `Providers` component that wraps the root layout (Pattern 1 above).
**Warning signs:** `"No nuqs adapter found"` error on first page load.
[VERIFIED: Context7 /47ng/nuqs]

### Pitfall 3: params Must Be Awaited (Next.js 15+)

**What goes wrong:** Code like `const { id } = params` in a page or layout causes a runtime error.
**Why it happens:** In Next.js 15+, `params` and `searchParams` are Promises, not plain objects.
**How to avoid:** Always `const { id } = await params` in async server components and layouts.
**Warning signs:** TypeScript will warn; runtime error if ignored.
[VERIFIED: Context7 /vercel/next.js/v16.2.2]

### Pitfall 4: In-Memory Store Resets on Every Route Handler Hot Reload

**What goes wrong:** In dev mode with Turbopack, Route Handler modules may re-initialize on code changes, resetting the module-scoped Map.
**Why it happens:** Turbopack HMR replaces modules, losing module-level state. Unlike the browser singleton pattern, server-side module state is not preserved across hot reloads.
**How to avoid:** This is expected and intentional (D-21: "restart = fresh demo state"). Document clearly in README. Add a dev-only "Reset Store" button that simply triggers a page reload if the client asks for a clean state mid-demo.
**Warning signs:** Mutations made in one browser tab disappear after any file save in the editor.
[ASSUMED — based on Next.js HMR behavior knowledge; verified indirectly via WebSearch finding module state persistence issues in Next.js]

### Pitfall 5: suppressHydrationWarning Required for Dark Mode

**What goes wrong:** next-themes causes a React hydration mismatch warning because the server renders without theme knowledge.
**Why it happens:** The server doesn't know which theme the client prefers until JS runs.
**How to avoid:** Add `suppressHydrationWarning` to the `<html>` element in root `layout.tsx`.
[VERIFIED: Context7 /pacocoursey/next-themes]

### Pitfall 6: Stitch Token Classes Left in Wireframe-Derived Markup

**What goes wrong:** If copy-pasting structural HTML from wireframes, Stitch class names like `bg-surface`, `text-on-surface`, `bg-surface-container-lowest` slip through. These render nothing (no Tailwind class match) and produce invisible content.
**Why it happens:** The wireframes use a custom design system not present in the radix-vega preset.
**How to avoid:** Never copy class names from wireframes. Use wireframes only for layout structure (grid, sections, order of elements, column names). Apply radix-vega tokens and shadcn component variants from scratch.
**Warning signs:** Content areas that are blank or invisible when inspected in browser DevTools show unrecognized utility classes.

### Pitfall 7: next@16.1.7 vs 16.2.3 Version Drift

**What goes wrong:** The preset scaffolds 16.1.7 but npm latest is 16.2.3. If D-03 requires bumping, mixing it with scaffold work in the same commit creates an unreviewed diff.
**Why it happens:** Preset pins a slightly older minor version.
**How to avoid:** Per D-03, do the bump as a single dedicated commit after scaffold. Check `next/dist/version.js` or `package.json` for what the preset produces, then compare release notes for 16.1.7 → 16.2.3 breaking changes before bumping.
[VERIFIED: npm registry shows 16.2.3 as latest]

### Pitfall 8: Three Third-Party Detail Files → One Dynamic Route

**What goes wrong:** The three wireframes (`third-party-detail-supplier.html`, `third-party-detail-garage.html`, `third-party-detail-finance.html`) could naively be implemented as three separate routes.
**Why it happens:** Each has distinct columns and sections.
**How to avoid:** D-14 specifies a single `/(app)/third-parties/[id]` route. Use the fixture `type` field to conditionally render the correct variant. All three wireframes should be read as IA references for their respective render branches.
