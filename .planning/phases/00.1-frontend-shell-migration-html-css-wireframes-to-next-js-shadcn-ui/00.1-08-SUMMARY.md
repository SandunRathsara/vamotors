# Phase 00.1 Plan 08: Governance Screens (Approvals, Users, Settings) Summary

Approvals workflow, Users & Roles management, and Settings page with all 4 card sections. Completes the Management and System sidebar sections.

## What Was Built

**Task 1 — Approvals:**
- Extended `GET /api/approvals` with category filter + sortBy/sortDir params
- Added `PATCH /api/approvals` for approve/reject actions
- Columns show Approve/Reject inline buttons for Pending rows; View-only for Approved/Rejected
- `ApprovalActionDialog` handles 4 modes with exact UI-SPEC destructive copy ("Cancel this advance without a refund?", "Cancel this advance and refund the customer?")
- Status and Category filter dropdowns in toolbar

**Task 2 — Users:**
- `GET/POST/PATCH /api/users` route
- Name column with Avatar initials + email sub-line
- Delete User ConfirmDialog with UI-SPEC copy: "This will permanently remove their access. This cannot be undone."
- AddUserDialog: Name, Email, Role (4 options), Password

**Task 3 — Settings:**
- `GET/PATCH /api/settings` route
- 4 Card sections: Company Profile, Currency Configuration, Advance Payment Tiers, System Preferences
- AdvanceTiersTable with inline-editable Validity Days and Label inputs
- Form prepopulates from API on mount; Save Changes triggers "Settings saved." toast

## Deviations from Plan

**1. [Rule 1 - Fix] Settings useEntityDetail → useEntityQuery**
- **Found during:** Task 3
- **Issue:** Plan specified `useEntityDetail('settings', '/api/settings', 'current')` which constructs URL `/api/settings/current` — but settings has no `[id]` route segment; `GET /api/settings` returns the full object directly.
- **Fix:** Used `useEntityQuery('settings', '/api/settings', {})` instead, which hits `GET /api/settings` correctly.
- **Files modified:** `components/settings/settings-form.tsx`

## Known Stubs

None — all data is wired to mock store. Settings prepopulates from `settingsFixture`, approvals and users from their respective fixture stores.
