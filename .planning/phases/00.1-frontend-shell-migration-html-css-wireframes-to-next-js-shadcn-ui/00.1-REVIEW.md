# Phase 00.1: Code Review Report

**Reviewed:** 2026-04-11
**Depth:** standard
**Files Reviewed:** 68
**Status:** issues_found

## Summary

This phase delivers a complete frontend shell using Next.js 15 App Router, shadcn/ui, TanStack Query, and mock data. The overall architecture is clean: Route Handlers are consistent, the `MockStore` generic is well-designed, Zod schemas are properly co-located, and the `useEntityQuery`/`useEntityMutation` hook abstractions reduce boilerplate effectively.

No critical issues were found. The warnings fall into four clusters: (1) a `notFound()` usage in Route Handlers that throws an unhandled HTML response rather than JSON, (2) the `MockStore` being a module-level singleton that will silently share state across serverless function instances in production (irrelevant for mock phase but a migration trap), (3) stale closure and missing-memoization bugs in component-level columns/callbacks, and (4) missing `'use client'` directives on two shared components. Info items are mostly cosmetic — unused `variant` prop, index-as-key in a few lists, minor label inconsistencies.

## Warnings

### WR-01: `notFound()` used inside Route Handlers returns HTML, not JSON

**Files:**
- `app/api/vehicles/[id]/route.ts:16`
- `app/api/sales/[id]/route.ts:15`
- `app/api/lease-deals/[id]/route.ts:14`

**Issue:** `notFound()` from `next/navigation` throws Next.js's special `NEXT_NOT_FOUND` error, which the framework catches and renders the nearest `not-found.tsx` page (HTML). When called inside a Route Handler, the client receives an HTML 404 page instead of a JSON error body. The sibling handlers (`customers/[id]`, `third-parties/[id]`) correctly return `NextResponse.json({ error: "..." }, { status: 404 })`. This inconsistency means `useEntityDetail` receives an HTML response, `res.json()` throws a parse error, and the UI surfaces a generic "something went wrong" error rather than a structured not-found state.

**Fix:**
```typescript
// Replace:
if (!vehicle) {
  notFound()
}

// With:
if (!vehicle) {
  return NextResponse.json({ error: "Vehicle not found" }, { status: 404 })
}
```

Apply the same pattern in `app/api/sales/[id]/route.ts` and `app/api/lease-deals/[id]/route.ts`.

### WR-02: `MockStore` instances are module-level singletons — serverless will not share state between invocations

**File:** `lib/mock-store.ts:135-148`

**Issue:** Each store is instantiated once at module load time. In Next.js development with hot-reload this works as expected. However, when the real database phase arrives, any developer who copies the singleton pattern without understanding its serverless behaviour will create a production bug: Vercel serverless functions do not share module-level memory across concurrent invocations. For the mock phase this is harmless. The risk is that this pattern gets copied when wiring up the real Prisma client — the CLAUDE.md already documents the Prisma singleton with `globalThis`, but the mock pattern is structurally similar enough to cause confusion.

**Fix:** Add a comment to the store instantiation block to clearly mark this as intentionally ephemeral for the mock phase:

```typescript
// ── Store instances (module-scoped singletons, MOCK ONLY) ─────────────────────
// These stores live in module memory and reset on each cold start.
// They are intentionally NOT backed by globalThis — do not carry this
// pattern forward when wiring up Prisma.
export const vehiclesStore = new MockStore<Vehicle>(vehicleFixtures)
// ...
```

### WR-03: Stale closure bug in `ThirdPartiesTable` — columns captured without `handleArchive` in deps

**File:** `components/third-parties/third-parties-table.tsx:85-89`

**Issue:** The `columns` memoization deliberately excludes `handleArchive` from the dependency array (suppressed with `eslint-disable`). The `handleArchive` callback closes over `setArchiveTarget`, which is stable from `useState`, so in practice this is safe. However, the explicit suppression comment is incorrect and fragile. The real dependency is the component-level `handleArchive` function, which is redefined on every render. Without `useCallback` on `handleArchive`, if `handleArchive` ever closes over anything that changes (e.g., after adding toast or router calls), the bug will silently manifest as stale data in the archive dialog.

**Fix:**
```typescript
// Wrap handleArchive in useCallback
const handleArchive = React.useCallback((tp: ThirdParty) => {
  setArchiveTarget(tp)
}, []) // setArchiveTarget is stable

const columns = React.useMemo(
  () => createThirdPartiesColumns(handleArchive),
  [handleArchive],
)
```

### WR-04: `ReconciliationTable` — `buildColumns` called on every render, recreating all column objects

**File:** `components/lease/reconciliation-table.tsx:196-199`

**Issue:** `buildColumns(...)` is called directly in the component body (line 196), not wrapped in `useMemo`. This recreates all column definition objects on every render. While TanStack Table does not break on this, it forces an internal re-reconciliation pass on every state update (filter changes, pagination, dialog open/close). The `onReconcile` callback passed to `buildColumns` also captures `setSelectedItem` and `setReceivedAmount`, so it is safe to memoize.

**Fix:**
```typescript
const columns = React.useMemo(
  () => buildColumns((item) => {
    setSelectedItem(item)
    setReceivedAmount(String(item.expectedCommission))
  }),
  [], // setSelectedItem and setReceivedAmount are stable
)
```

### WR-05: `SettingsForm` — `hydrated` ref pattern will not re-sync if settings are refetched

**File:** `components/settings/settings-form.tsx:54-71`

**Issue:** The `hydrated` ref (line 54) blocks re-hydration after the first load. If TanStack Query refetches (e.g., another tab saves, stale time expires, or the user manually invalidates), the form will silently ignore the updated server data. For a settings form this is an unlikely problem in the mock phase, but it is a broken pattern that will cause issues once real persistence is added — the user will see a conflict between their in-flight edits and a background refetch with no merge strategy.

**Fix:** Remove the ref and replace with `useEffect` that only runs on mount (empty dep array is not needed — instead guard on the specific initial load):

```typescript
// Option A: populate once on first successful load using isSuccess
const { data: settings, isLoading, isSuccess } = useEntityQuery<Settings>(...)

const [initialised, setInitialised] = React.useState(false)
React.useEffect(() => {
  if (isSuccess && settings && !initialised) {
    setInitialised(true)
    setCompanyName(settings.companyName)
    // ...
  }
}, [isSuccess, settings, initialised])
```

This is semantically identical to the ref approach but explicit in the React model. Document that background refetch will not overwrite in-flight edits.

### WR-07: `CommandMenu` keyboard shortcut toggles on stale `open` value

**File:** `components/layout/command-menu.tsx:27-35`

**Issue:** The `useEffect` at line 27 captures `open` in its closure for the toggle logic (`setOpen(!open)`). Because `open` is included in the dep array, the handler is re-registered on every open/close change. This is correct for preventing stale closure on the value, but the pattern creates a minor flicker risk: if two rapid Cmd+K presses arrive before the state update propagates, both may toggle based on the same stale `open` value. This is a race condition in the React update cycle.

**Fix:** Use the functional updater form to avoid reading `open` from the closure:

```typescript
function handleKeyDown(e: KeyboardEvent) {
  if ((e.metaKey || e.ctrlKey) && e.key === "k") {
    e.preventDefault()
    setOpen((prev) => !prev)
  }
}
// Remove `open` from the dep array — handler no longer reads it
document.addEventListener("keydown", handleKeyDown)
return () => document.removeEventListener("keydown", handleKeyDown)
// eslint-disable-next-line react-hooks/exhaustive-deps
}, [setOpen])
```

### WR-08: `CurrencyDisplay` and `StatusBadge` missing `'use client'` directive

**Files:**
- `components/shared/currency-display.tsx`
- `components/shared/status-badge.tsx`

**Issue:** Both components render JSX and are imported by client components. Neither has a `'use client'` directive. In Next.js 15 App Router, components without `'use client'` are treated as Server Components by default. These two are pure display components with no server-side logic (no DB access, no `server-only` imports). When imported by a client component, they are implicitly treated as client components via the client component tree, so the app does not currently break. However, this implicit promotion is a subtle dependency on import order. If either is ever imported from a Server Component context (e.g., a future RSC page renders a StatusBadge inline), it will fail silently or throw because JSX in a server component context behaves differently when the import graph is ambiguous.

**Fix:** Add `'use client'` to both files since they are pure presentation components intended for the client tree:

```typescript
// components/shared/currency-display.tsx
"use client"

import { cn } from "@/lib/utils"
// ...
```

```typescript
// components/shared/status-badge.tsx
"use client"

import { Badge } from "@/components/ui/badge"
// ...
```

### WR-09: `dashboard/route.ts` monthly sales chart counts sales from all years for same month index

**File:** `app/api/dashboard/route.ts:26-32`

**Issue:** The `monthlySales` array is built by matching `d.getMonth() === idx` (line 29), which compares only the month number (0–11) without filtering by year. A sale from January 2023 and a sale from January 2025 both increment the "Jan" bucket. The result is a cumulative multi-year count displayed as if it were a single year's data, which is misleading for a "units sold per month" chart.

**Fix:** Filter by the current year before building the monthly buckets:

```typescript
const currentYear = new Date().getFullYear()
const monthlySales = MONTHS.map((month, idx) => {
  const count = sales.filter((s) => {
    const d = new Date(s.date)
    return d.getFullYear() === currentYear && d.getMonth() === idx
  }).length
  return { month, count }
})
```

### IN-05: `RecordSaleDialog` — sale price parsed with `parseFloat` then multiplied by 100, inconsistent with storage convention

**File:** `components/sales/record-sale-dialog.tsx:56`

**Issue:** The sale price input uses `type="number"` with `step="0.01"`, and the value is submitted as `Math.round(parseFloat(salePrice) * 100)`. This is the correct pattern for the "amount entered in LKR units, stored in cents" convention documented in CLAUDE.md. However, the input label says "Sale Price (LKR)" which implies the user enters whole LKR units — but the `step="0.01"` allows decimal entry, creating ambiguity about what the user should type (e.g., does "3500000" mean 35,000.00 LKR or 35,000,000 LKR?).

The `LeaseDealsTable` new deal form at line 188 labels the loan amount as "Loan Amount (LKR cents)" (i.e., the user is expected to type raw cents), but then the same value is stored directly without the `* 100` conversion. These two forms are inconsistent in their conventions.

**Fix:** Standardize across forms:
- Remove `step="0.01"` from the sale price input (users should enter whole LKR).
- Change the `LeaseDealsTable` label from "Loan Amount (LKR cents)" to "Loan Amount (LKR)" and apply `* 100` on submit, matching `RecordSaleDialog`.

### IN-06: `FileDropzone` — blob URL cleanup only runs on unmount, not on individual file removal

**File:** `components/shared/file-dropzone.tsx:57-62`

**Issue:** The `useEffect` cleanup at line 57 revokes all blob URLs when the component unmounts. Individual file removals via `removeFile` (line 116-121) do correctly call `URL.revokeObjectURL`. However, the `useEffect` cleanup captures `files` at mount time (the dep array is intentionally empty with a suppressed lint warning). If files are added but the component never unmounts (e.g., it is inside a persistent form), and the parent resets the `files` state to `[]` without calling `removeFile` for each entry (e.g., after a successful form submit), the blob URLs from those files will leak for the session lifetime.

**Fix:** Either document that the parent must call `removeFile` for each file before clearing, or change the cleanup strategy to track all URLs in a ref:

```typescript
const blobUrlsRef = React.useRef<Set<string>>(new Set())

// In processFiles, after creating previews:
newPreviews.forEach((p) => blobUrlsRef.current.add(p.previewUrl))

// In removeFile:
URL.revokeObjectURL(target.previewUrl)
blobUrlsRef.current.delete(target.previewUrl)

// Cleanup on unmount:
React.useEffect(() => {
  return () => {
    blobUrlsRef.current.forEach((url) => URL.revokeObjectURL(url))
  }
}, []) // Safe because we track via ref, not state
```
